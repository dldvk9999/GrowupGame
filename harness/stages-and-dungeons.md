# 스테이지 & 던전

관련 파일: `stages.js`, `stageStory.js`, `StageSelect.jsx`, `dungeonStages.js`, `dungeon.js`, `DungeonSelect.jsx`, `DungeonBattle.jsx`, `jobDungeon.js`, `jobDungeonApi.js`, `JobDungeonBattle.jsx`

> 던전 탭에는 경험치/골드/전직 던전 외에 **월드보스** 서브탭도 있음 — 전체 유저 공용 주간 레이드라 별도 문서 [`world-boss.md`](./world-boss.md)에 정리함.

## 스테이지 시스템

- **100챕터 × 10스테이지 = 1000스테이지**, `stage_id`는 1~1000 순번(`toStageIndex(chapter, stage)`)
- 각 챕터 10번째 스테이지는 **보스**(체력/공격력 배율 ↑)
- 챕터별 속성은 `fire→water→grass` 3개씩 순환. 원래 텍스트 배지가 22px 원형 안에서 개행이 어색하게 깨져서, 텍스트 대신 아이콘(🔥/💧/🌿)으로 교체(`StageSelect.jsx`의 `ELEMENT_ICON`)
- 몬스터 스탯은 전체 진행도(index)에 비례해 절차적으로 스케일링(`getStageEnemy`)
- **스토리 진행**: 새 챕터 첫 진입 시 `ChapterStory` 배너(`getChapterStory`, 화면 중앙), 서브스테이지 진입마다 전투 로그에 짧은 플레이버 텍스트(`getStageFlavor`)
- **스테이지 잠금 해제**: `stage_id===1`이거나 이전 스테이지 클리어 → 오픈. 클리어한 스테이지는 언제든 재도전 가능
- **스테이지 선택 UI**(`StageSelect.jsx`): 챕터를 좌우로 스와이프하는 카드 캐러셀. 카드에 대표 몬스터/챕터명/속성/진행도/스토리 요약, 선택하면 하단에 10개 서브스테이지 그리드. 잠긴 챕터는 회색+자물쇠, 현재 위치는 "현재" 뱃지

### 난이도 공식

```
index = (chapter-1)*10 + stage
isBoss = (stage === 10)
chapterStep = 1 + (chapter-1)*0.05              // 챕터(10스테이지) 단위 계단식 상승
midChapterStep = stage >= 5 ? 1.15 : 1          // 같은 챕터 안에서도 5번째 스테이지부터 한 단계 더 상승
stepMultiplier = chapterStep * midChapterStep
hp  = round((30 + index*11.0*(isBoss?3.6:1)) * stepMultiplier)
atk = round((4 + index*1.05*(isBoss?2.9:1)) * stepMultiplier)
def = round((3 + index*0.6*(isBoss?2.7:1)) * stepMultiplier)
```

- 030에서 **대폭 상향**(전직을 거듭할수록 스테이지가 너무 쉬워진다는 피드백) — 계수 자체가 커졌고, 5스테이지째부터도 한 단계 더 강해지는 계단(`midChapterStep`) 추가. 챕터가 바뀔 때(10스테이지)도 `chapterStep`이 한 단계 더 올라 "5마다 한 번, 10마다 한 번 더" 이중 계단 구조
- 040에서 **일반(비보스) 몹만 추가로 1.8배 상향**(`NORMAL_MONSTER_BOOST`) — 보스 배율에 비해 일반 몹이 너무 물렁해서(고레벨 기준 스킬 1방에 끝남) 보스는 그대로 두고 일반 몹만 별도 상향. `calc_stage_gold`도 동일 배율 동기화
- `def`는 방어력 신설 시 추가, 몬스터/보스/일반던전보스/전직던전보스 전부 적용
- 데미지 계산은 [`combat.md`](./combat.md)의 `mitigateDamage`로 경감
- **적 공격 텀이 스테이지 진행도에 비례해 빨라짐 + 공격력 계수 상향 (사용자 요청)**: 기존엔 공격 텀이 항상 고정(1.9초)이라 스테이지가 오를수록 "한 방이 세질" 뿐 "더 자주 맞진" 않았음 — `getEnemyAttackInterval(stageIndex, isBoss)`가 `max(700, round((1900 - stageIndex*1.1) * (isBoss ? 0.82 : 1)))`로 스테이지 인덱스에 비례해 공격 텀을 줄임(최저 700ms, 보스는 18% 더 빠름). 공격력 계수도 `0.85→1.05`(보스는 `2.6→2.9`)로 상향해서, "더 세게 + 더 자주" 맞는 만큼 방어구/신발(DEF/HP)의 상대적 가치가 커지도록 함. 던전(경험치/골드)도 동일 취지로 `getDungeonAttackInterval(stage) = max(700, round(1900 - stage*2.2))` 신설 + 공격력 계수 `13→15` 상향(`dungeonStages.js`) — 던전은 매 층이 이미 "보스"라 별도 보스 배율 없이 층수만으로 감소. 무한의 탑/월드보스/전직던전은 각자 독립된 스탯 공식을 쓰고 있어 이번 변경 범위 밖(의도적으로 손 안 댐)
- **체력/방어력 추가 대폭 상향 (migration 133, 신규, 사용자 요청 — "클리어 난이도를 대폭 상향")**: 앞선 상향(공격력/공격속도)이 "더 아프게 맞는" 방향이었다면, 이번엔 "잡기 힘들고(HP 계수 7.5→11.0, 보스 3.0→3.6) 내 공격이 잘 안 먹히는(DEF 계수 0.4→0.6, 보스 2.2→2.7)" 방향으로 클리어 자체를 어렵게 만듦. 시뮬레이션 기준 HP/DEF 전 구간 약 1.7~1.85배 상승. 공격력은 지난 상향에서 이미 크게 올렸으니 이번엔 안 건드림
- ⚠️ 이 상향으로 최후반(챕터100/스테이지10) 보스 골드가 약 69만까지 치솟아 `add_gold` 상한도 400000→**1,000,000**으로 재상향(030)
- ⚠️ **[예방적 발견/수정] `clear_stage` 골드 클램프 누락(migration 133)**: 133의 HP 상향으로 `calc_stage_gold`가 계산하는 최후반 보스 골드가 약 122만까지 치솟았는데, `clear_stage`(096)의 최종 100만 클램프가 **"정예 몬스터(8% 확률, 2배)" 분기에만 걸려있고 평상시(92%) 경로엔 아예 없었음** — 배포됐다면 "가장 어려운 마지막 보스를 깨면 골드 보상이 `add_gold` 상한을 넘어 크래시로 아무것도 못 받는" 심각한 회귀가 났을 것(117/118/121과 같은 클래스의 함정, 배포 전 시뮬레이션 검증 중 발견). 정예 여부와 무관하게 무조건 한 번 더 클램프하도록 수정
- 너무 세거나 약하면 `stages.js`/`dungeonStages.js`/`jobDungeon.js`의 계수를 조정하면 됨

### 보상 공식

```
expReward  = round(hp * (isBoss ? 1.15 : 0.65))
goldReward = (round(hp * (isBoss ? 0.9 : 0.4)) + stage*2) * 5
```

서버 `calc_stage_gold`도 동일 공식을 SQL로 반영(골드는 서버 최종 계산, `def`는 골드 공식에 미영향).

⚠️ **경험치 하향 이력(사용자 요청)**: 원래 배율(일반몹 0.85 / 보스 1.5)에서 각각 0.65 / 1.15로 하향(약 23~24% 감소). 골드는 유지, 경험치만 조정. 레벨업/경험치는 서버 검증 없이 클라이언트 계산값을 `save_monster_growth`로 그대로 저장하는 구조([`security.md`](./security.md))라, 클라이언트 배율만 조정하면 바로 반영됨.

### 자동사냥으로 스테이지 자동 진행 ("🔁 자동사냥", 신규, 사용자 요청)

기존 자동사냥(idle 모드)은 공격력 0인 가짜 "필드 몬스터"(`getIdleMonster`)를 farming하는 것뿐이라, 실제 스테이지를 깨려면 항상 수동으로 "도전하기"를 눌러야 했음. `BattleScreen.jsx`에 "🔁 자동사냥" 버튼을 추가해서, 켜두면 **진짜 스테이지 전투(challenge 모드, `getStageEnemy` 동일 스탯)를 자동으로 계속 진행**함.

- `autoPush` 상태는 `App.jsx`(부모)에 있음 — `BattleScreen`이 `key={currentStageIndex}`라 스테이지가 바뀔 때마다 **컴포넌트 전체가 리마운트**되기 때문에, 로컬 state로 두면 스테이지 넘어갈 때마다 꺼져버림. 부모에 올려서 리마운트돼도 유지되게 함
- **자동 스킬 사용**: 550ms마다 장착 스킬 중 쿨다운 안 걸린 걸 찾아 자동으로 사용(데미지 타입 우선 순회) — 사람이 스킬 버튼을 순서대로 누르는 걸 흉내냄
- **승리 시**: 0.9초 뒤 자동으로 `onAdvance()`(다음 스테이지로) 호출 → `key` 변경으로 새 `BattleScreen`이 마운트되고, `autoPush`가 여전히 true라 그 스테이지에서도 즉시 자동 도전이 시작됨 — 이런 식으로 계속 스테이지를 밀고 나감
- **패배 시**: 자동사냥이 그 자리에서 멈춤(사용자 요청: "몬스터를 잡지 못한다면 이전 스테이지에서 계속 반복") — `onAutoPushStop()`으로 부모의 `autoPush`를 꺼서 더 이상 자동으로 재도전하지 않게 하고, 1.4초 뒤 `backToIdle()`로 원래의 약한 필드몬스터 파밍으로 돌아감. 경험치/골드는 계속 조금씩 벌리는 상태 유지. **"🔁 자동사냥" 버튼을 다시 누르면 그 스테이지부터 재도전** 재개
- 챕터 첫 진입 시 스토리 팝업이나 인터루드([`story.md`](./story.md))가 뜨면 `BattleScreen`이 언마운트되지만, 팝업을 닫고 돌아오면 `currentStageIndex`가 이미 갱신돼있고 `autoPush`도 부모에 그대로 남아있어서 자동으로 이어서 진행됨(추가 처리 불필요)
- 몬스터 스탯은 100% 실제 스테이지 전투와 동일한 `getStageEnemy`를 그대로 재사용 — 별도 "자동사냥 전용 난이도" 같은 건 없음(사용자 요청: "몬스터의 능력치는 실제 스테이지 전투와 동일하게")

### 자동사냥 보상

```
hp = max(10, round(8 + chapter*2.0 + playerLevel*3.0))
expReward  = max(1, round(hp*0.19*0.5))
goldReward = max(5, round(hp*0.15)*5*8)
```

- 챕터/레벨 가중치를 대폭 올려(기존 0.6/0.8 → 2.0/3.0) **레벨/진행 챕터가 높을수록 자동사냥 보상이 눈에 띄게 커짐**
- ⚠️ **경험치 절반 감소(사용자 요청)**: `expReward`에 ×0.5 추가 — 경험치만 하향(레벨업 속도 조절 목적으로 추정). 스테이지 도전(챌린지 모드) 경험치는 별도 공식이라 영향 없음.
- ⚠️ **골드도 절반 감소(migration 134, 신규, 사용자 요청)**: `calc_idle_gold`(서버, `grant_idle_reward`/오프라인보상/주말보너스/황금몬스터/유물골드%/시즌이벤트 전부 이 값을 기반으로 배율만 얹는 구조)에 최종 ×0.5 반영 — **단일 진실 공급원 하나만 고쳐서 위 보너스들 전부 비율 그대로 유지하며 일괄 절반 감소**됨(오프라인보상은 원래도 "온라인 대비 50%"라 결과적으로 온라인의 25% 수준이 됨, 의도된 동작). 클라이언트 `stages.js`의 `goldReward`(자동사냥 로그 표시용 복사본)도 동일하게 반영 안 하면 "로그엔 실제 받은 것의 2배가 찍히는" 표시 불일치가 생겨서 같이 수정함(경험치 절반감소 때 `idleTimeEstimate.js`를 놓칠 뻔했던 것과 같은 클래스의 이중 공식 관리 주의사항)
- 자동사냥 처치 텀은 **1.5초**
- 서버 `calc_idle_gold`도 동일 공식 반영
- ⚠️ **경험치 하향 이력(사용자 요청)**: 원래 배율(0.25)에서 0.19로 하향(약 24% 감소). 골드는 유지
- ⚠️ **버그 수정 이력(037)**: `grant_idle_reward(p_chapter, p_player_level)`가 두 파라미터를 클라이언트가 보낸 값 그대로 신뢰해서, devtools로 과장된 값을 반복 호출하면 진행도와 무관하게 골드를 무제한 파밍할 수 있었음. 서버가 `owned_monsters`(실제 레벨)/`stage_progress`(실제 최고 챕터)에서 직접 계산하도록 수정 — 함수 시그니처는 그대로라 클라이언트 변경 불필요
- **🌟 황금 몬스터 이벤트(062, 신규 콘텐츠)**: 자동사냥 처치마다 서버가 5% 확률(`random() < 0.05`)을 자체 판정, 당첨되면 골드 보상 **3배**. 서버가 스스로 판정하고 결과(`is_golden`)만 반환하는 구조라 클라이언트 조작 불가(2.5초 최소 간격과 결합해 평균 약 50초에 1회꼴). "🌟 황금 몬스터 발견! 골드 3배 획득" 토스트. `grant_idle_reward`의 반환 타입이 `integer` → `table(gold, is_golden)`로 바뀌어 `DROP FUNCTION` 선행 필요(062에 포함)
- **🎉 주말 골드 1.5배 이벤트 (migration 105, 신규 콘텐츠)**: 한국시간 기준 토·일요일엔 `grant_idle_reward`가 계산된 골드에 1.5배를 곱함(황금 몬스터 3배와 중첩 가능, 최종 100만 골드로 sanity 클램프). 매 틱(1.5초)마다 토스트를 띄우면 스팸이 되므로, 실제 배율 적용과는 별개로 **로그인 시 하루 1번**만(`weekendBonus.js`, `dailyQuote.js`와 동일한 localStorage 날짜추적 패턴) "🎉 주말 이벤트! 자동사냥 골드 1.5배 진행중" 안내. 반환 컬럼 구성이 그대로라 `DROP FUNCTION` 불필요
  - ⚠️ **버그 수정 이력(118)**: 105 최초 배포 시 이 클램프가 500만으로 잘못 설정돼있었음(`add_gold`의 실제 상한은 100만, 030). 레벨 상한이 없는 게임이라 대략 레벨 12,000대 이상에서 주말+황금몬스터 배율(약 4.5배)이 겹치면 골드가 100만을 넘어 `add_gold`가 크래시할 수 있던 잠재 버그. 117(PvP 보상 오버플로우 예방수정) 작업 직후 "다른 곳도 확인해보자"며 전체 클램프 값을 grep으로 재검토하다 발견 — 다른 7곳(071/072/075/080/087/096/098/100)은 전부 정확히 100만으로 돼있었는데 105만 유일하게 틀려 있었음. 100만으로 정정

### 시즌 한정 이벤트 프레임워크 (migration 123, 신규 콘텐츠, 사용자 요청)

짧은 기간(3~7일)만 켜두는 이벤트를 만들어달라는 요청 반영. 별도 테이블/상태 없이 `grant_idle_reward` 안에서 **날짜 범위만 하드코딩으로 판정**하는 가장 단순한 형태 — 105(주말)/118/120(유물)과 같은 골드 배율 파이프라인에 자연스럽게 얹음(새 통합 지점 안 만듦).

### 오프라인(방치) 골드 보상 (migration 103, 106에서 취약점 수정)

"자리를 비운 사이에도 자란다"는 키우기 게임의 핵심 기대치를 실제로 구현한 신규 콘텐츠. 기존엔 앱을 켜둔 상태에서만 위 자동사냥 골드가 나갔는데, 로그인 시 `claim_offline_gold_reward()`가 마지막 접속 이후 경과 시간만큼 골드를 즉시 지급함.

- `profiles.last_offline_claim_at`(106 신규 컬럼) 기준. **이 함수 자신이 매 호출마다 체크포인트를 `now()`로 갱신**함(102의 `last_login_at`과는 완전히 별개 컬럼 — grant_idle_reward의 `last_idle_reward_at` 자체갱신과 동일한 패턴)
- ⚠️ **취약점 수정 이력(106)**: 103 최초 버전은 이 함수가 `last_login_at`을 "읽기만 하고 갱신 안 함"(갱신은 102의 복귀 보상 함수가 전담) 구조였음. 그런데 이러면 devtools로 `claim_offline_gold_reward()`만 반복 호출 시 체크포인트가 절대 안 움직여서 같은 오프라인 보상을 **무한 반복 지급**받을 수 있었음(치명적 취약점, 자세한 배경은 [`security.md`](./security.md)). 전용 컬럼 신설 + 자체갱신 + `for update` 락으로 수정
- 방치 5분 미만이면 지급 안 함(짧은 재접속/반복 호출마다 팝업·지급 뜨는 것 방지), 최대 **10시간 분량**까지만 인정(137에서 2시간→10시간 상향, 사용자 요청)
- 골드량 = `calc_idle_gold(chapter, level) × (경과초/2.5) × 0.5` — 온라인 자동사냥 대비 **50% 효율**로 설계해 "그래도 직접 하는 게 더 낫다"는 균형 유지
- 활성 몬스터가 없는 상태(스타터 계약 전)면 조용히 0 반환
- `claim_dungeon_reward`류와 달리 우편함이 아니라 **즉시 지급**(로그인 직후 "💤 자리를 비운 N분 동안 골드 XX를 벌어왔어요!" 토스트+효과음) — 오프라인 성장 보상은 즉각적인 보상감이 리텐션에 더 효과적이라는 판단
- ⚠️ **[예방적 발견/수정, migration 137] 최종 클램프 누락**: `v_gold`를 `add_gold`에 넘기기 전 클램프가 103/106 어디에도 없었음. 계산해보니 **기존 2시간 설정 그대로도** 챕터30·레벨55 정도의 중견 유저면 이미 100만을 넘어 `add_gold`가 크래시할 수 있는 상태였음(이번 10시간 상향과 무관하게 원래도 잠재됐던 버그, 배포 전 시뮬레이션 중 발견 — 117/118/121/133과 같은 클래스). `least(v_gold, 1000000)` 추가로 수정

- ⚠️ `claim_dungeon_reward`는 "세션 생성 후 최소 2초" 경과해야 클레임 가능(037) — "실제로 이겼는지" 검증은 아니고, 입장 직후 바로 클레임하는 가장 단순한 우회만 막는 부분적 완화책([`security.md`](./security.md))

## 챕터 클리어 축하 보너스 (migration 063, 신규 콘텐츠)

각 챕터의 마지막 스테이지(10번째, 보스)를 **처음** 클리어하면 `챕터번호 × 5000` 골드가 축하 우편으로 자동 발송(챕터10=5만, 챕터100=50만, 100만 상한 이내로 안전).

- `clear_stage(p_stage_id)`가 진행도를 기록하기 **직전** 기존 `cleared` 상태를 조회해 "이번이 첫 클리어인지" 판정 — 재도전 시 중복 지급 안 됨
- `mails.source_key`(`'chapter_clear_' || 챕터번호`)로 이중 안전장치
- 반환값(`integer`, 기존 스테이지 골드)은 그대로라 DROP 불필요 — 챕터 보너스는 별도 우편으로만 지급
- `App.jsx`의 `handleClear`가 `clearedStageIds`에 없던 스테이지 + 10의 배수 조합으로 "챕터 보스 첫 클리어"를 낙관적 판단해 즉시 뱃지/토스트 표시(정확한 값은 우편함에서 항상 재확인되므로 지급엔 영향 없음)

## ⚠️ 긴급 버그 수정: 던전 완주 축하 보너스가 10층에서 잘못 발송됨 (migration 086)

079에서 던전 최고층을 10→500으로 올렸는데, `claim_dungeon_reward`의 "완주 축하" 판정(`v_session.stage = 10`)은 그대로 남아있어서, 실제로는 500층이 진짜 완주인데 10층 클리어 시점에 잘못 발송되고 있었음(500층 관련 후속 작업 중 재검토하다 발견). 086에서 `stage = 500`으로 정정, 문구도 "500층까지 전부 클리어"로 수정, 보상도 2만→10만 골드로 상향(500층이 훨씬 오래 걸리므로). diff로 판정조건/문구/금액 외 로직은 보존됨을 확인.

## 던전 정예 몬스터 이벤트 (migration 087, 신규 던전 콘텐츠)

황금 몬스터(062)와 동일한 설계를 던전에도 적용. 던전 클리어마다 서버가 **8% 확률**을 자체 판정, 당첨되면 골드 **2배**(100만 상한 클램프 유지). 서버가 스스로 판정하고 결과(`is_elite`)만 반환해 클라이언트 조작 불가. `claim_dungeon_reward`의 반환 타입이 `integer` → `table(gold, is_elite)`로 바뀌어 DROP FUNCTION 선행 필요(087에 포함). 당첨 시 "👑 정예 몬스터였어요! 골드 2배 획득" 토스트 + 황금몬스터와 동일 사운드.

## 던전 깊이 마일스톤 업적 (migration 088, 신규 던전 콘텐츠)

"던전 탐험가"(100층)/"던전 정복자"(300층)/"심연의 지배자"(500층 완주) — 500층 확장과 연동되는 진행 목표. 경험치/골드 던전 **둘 중 하나라도** 해당 층수에 도달하면 인정(`bool_or`로 통합 판정, 두 던전 모두 파야 하는 부담 방지). CASE만 추가, DROP 불필요, diff/36개 키 재검증 완료.

## 던전 상대 미리보기 (신규 콘텐츠, 사용자 경험 개선)

무한의 탑의 "다음 상대 미리보기"와 동일한 목적으로 던전 카드에도 도전 몬스터의 HP/ATK/DEF 표시(`getDungeonStage`가 반환하는 데이터 재사용, 새 계산 없음). 500층까지 열린 지금 특히 유용함. 무한의 탑의 `tower-opponent-stats` CSS 클래스 재사용(스타일 통일). 500층 극단값(HP 385만/ATK 14만/DEF 5만) 모바일 넘침 없음 재검증 완료.

## 전직 던전 카드에도 ATK/DEF 표시 (일관성 개선)

기존엔 HP만 표시했는데, 다른 던전들에 상대 미리보기를 추가한 김에 전직 던전 카드에도 ⚔️ATK/🛡️DEF를 추가해 일관성을 맞춤. `JOB_DUNGEON_BOSS`의 기존 데이터 재사용, 새 계산 없음. 모바일 넘침 재검증 완료.

## 스테이지 도전 정예 몬스터 이벤트 (migration 096, 신규 콘텐츠)

황금 몬스터(062)/던전 정예몬스터(087)에 이어 **스테이지 도전(challenge) 모드에도 동일한 8% 확률 골드 2배 이벤트**를 적용해 세 게임모드의 랜덤 보너스 일관성을 맞춤. `clear_stage`의 반환 타입이 `integer` → `table(gold, is_elite)`로 바뀌어 DROP FUNCTION 선행 필요(096에 포함). 챕터 보스 첫 클리어 보너스(063, 별도 우편)는 영향 없음 — 정예 이벤트와 챕터 보너스가 같은 판에 동시에 뜨면 챕터 축하 토스트를 우선 표시(더 큰 이벤트). diff로 스테이지 진행도 검증/챕터 보너스 로직 보존 확인.

⚠️ **자동사냥(5%)과 던전/스테이지(8%)의 확률이 다른 건 버그가 아니라 의도적 설계**: 자동사냥은 2.5초 최소 간격으로 매우 자주 반복돼(평균 약 50초에 1회) 확률이 낮아도 체감 빈도가 충분하지만, 던전/스테이지는 하루 3회 또는 진행형으로 훨씬 드물게 발생해 확률을 높여 "가끔 만나는 특별함"을 비슷하게 맞춤. **향후 반복 콘텐츠에 정예/황금 이벤트를 추가할 땐 그 콘텐츠의 반복 빈도를 먼저 고려해 확률을 정할 것**(기존 수치 무조건 복사하지 말 것).

## 던전 화면에 업적 마일스톤 진행률 안내 추가 (신규 콘텐츠)

던전 카드에 "🏅 다음 업적까지 N층 남음"(100/300/500층 깊이 업적, 088 연동) 표시. 실제 판정은 exp/gold 던전 둘 중 최댓값 기준이지만, 화면은 "지금 보는 타입" 기준으로 근사 안내(정확한 판정은 서버 재검증, 여긴 순수 안내용). 이미 있는 `clearedStage`만 계산, 새 서버 호출 없음.

## 던전 일일 콤보 보너스 (migration 098, 신규 던전 콘텐츠)

하루 3회 던전 입장권을 **전부 클리어**(입장만이 아니라 실제 승리+수령)하면 콤보 보너스 8000골드를 즉시 지급. "입장 횟수"(패배해도 증가)가 아니라 **"오늘 실제로 클리어해서 수령한 세션 수"**(`dungeon_sessions`에서 `claimed = true`인 오늘자 행 개수)로 정확히 판정. 매일 오전 8시(서울시간) 기준 하루 구분(`use_dungeon_attempt`와 동일 공식).

- `claim_dungeon_reward`의 반환 타입이 `table(gold, is_elite)` → `table(gold, is_elite, combo_bonus)`로 바뀌어 DROP FUNCTION 선행 필요(098에 포함)
- diff로 087 대비 콤보 보너스 로직만 추가되고 정예이벤트/완주보너스/스테이지진행도는 보존됨을 확인
- 완주보너스(우편)/정예이벤트(즉시 골드)/콤보보너스(즉시 골드)가 같은 판에 동시 발생해도 각각 독립적인 `add_gold` 호출이라 100만 상한 초과 문제 없음
- 콤보 보너스 당첨 시 정예 몬스터 토스트보다 우선 표시(더 큰 이벤트), 신기록 사운드 재생

## 이번 주 행운의 던전 (migration 100, 신규 던전 콘텐츠)

최신 방치형/아이들 게임의 "주간 로테이션 이벤트" 트렌드를 안전하게 적용. 매주 서버가 **결정론적으로**(별도 상태 테이블 없이 `extract(week from now())`의 홀짝만으로) 경험치/골드 던전 중 하나를 "이번 주 행운의 던전"으로 선정 — 클리어 시 골드 **1.5배**. 저장소/주간 초기화 작업 불필요, 월요일이 되면 자동으로 다음 주차. 다른 유저와도 항상 동일한 결과라 조작 불가.

- `claim_dungeon_reward`의 반환 타입에 `is_lucky_week`가 추가돼 DROP FUNCTION 선행 필요(100에 포함)
- 던전 선택 화면에서 미리 안내할 수 있도록 `fetch_lucky_dungeon_type()`(순수 계산, `auth.uid()` 불필요) 조회 전용 함수를 별도로 만듦 — **판정식이 두 함수에 각각 하드코딩돼있어 하나를 바꾸면 반드시 둘 다 같이 바꿔야 함**

## 요일별 던전 보너스 (migration 121, 신규 던전 콘텐츠, 사용자 요청)

주간 행운의 던전이 "이번 주 내내" 단위라면, 이건 "오늘 하루만" 적용되는 일간 보너스 — 매일 바뀌는 느낌을 주기 위해 추가. 한국시간 기준 요일로 결정론적 판정(별도 테이블 없음, 100과 동일한 설계 철학):

- **월/목**: 골드 던전 +30% / **화/금**: 경험치 던전 +30% / **수/토/일**: 요일 보너스 없음(주말은 이미 자동사냥 골드 1.5배가 따로 있어서(105) 매일 중첩시키지 않으려고 의도적으로 제외)
- 행운(1.5배)×요일(1.3배)×정예(2배) 전부 겹치면 최대 3.9배까지 가능 — 곱한 뒤에도 기존 100만 골드 클램프가 그대로 적용되는지 재확인 완료(어차피 `calc_dungeon_gold` 자체 상한도 100만이라 여러 배율이 겹쳐도 안전)
- `claim_dungeon_reward`의 반환 타입에 `is_daily_bonus` 추가돼 DROP FUNCTION 선행 필요(121에 포함), `fetch_daily_dungeon_bonus_type()` 조회 전용 함수 신설 — 마찬가지로 판정식이 두 함수에 하드코딩돼있어 하나 바꾸면 둘 다 같이 바꿔야 함
- 던전 선택 화면에 "📅 오늘은 OO 던전에 요일 보너스가 붙어요!" 배너, 클리어 시 "📅 오늘의 요일 보너스! 골드 1.3배 획득" 토스트(행운/정예 보너스가 없을 때만 표시 — 한 번의 클리어에 여러 배율이 겹쳐도 토스트는 우선순위 하나만)
- diff로 098 대비 행운의던전 로직만 추가되고 나머지는 전부 보존됨을 확인

⚠️ **긴급 클램프 버그 발견/즉시 수정**: `calc_dungeon_gold`(080에서 이미 100만 클램프됨)에 행운(1.5배)/정예(2배) 배율을 곱하는 순서상, "행운만 걸리고 정예는 안 걸린 경우"엔 클램프를 안 거치는 경로가 있어 **최대 150만 골드까지 나올 수 있었음**(500층 던전 확장 때 발견했던 것과 같은 유형의 실수 재발). 배포 전 시뮬레이션 검증 중 발견해서, 모든 배율을 곱한 뒤 **마지막에 한 번 더 `least(1000000, ...)`로 무조건 클램프**하도록 수정. **교훈**: 여러 배율을 순차적으로 곱하는 로직은 각 배율마다 개별 클램프를 넣기보다 모든 배율 적용 직후 최종 클램프를 한 번만 걸어야 조합 케이스를 놓치지 않음.

## 파견(원정) 시스템 (migration 122, 신규 콘텐츠, 사용자 요청)

무한원정대류 방치형 게임에 흔한 "몬스터를 잠깐 보내두고 나중에 받으러 온다" 패턴. 던전 탭의 6번째 서브탭 "🧭 파견"에서 짧은(30분)/중간(4시간)/긴(12시간) 중 하나를 선택해 시작하면, 시간이 지난 뒤 골드를 수령할 수 있음.

- **오프라인 골드 보상(103/106)과 역할 분리**: 오프라인 보상은 "앱을 닫고 있던 짧은~중간 시간"(최대 2시간, 50% 효율)을 커버하고, 파견은 "몇 시간 뒤 다시 켤 생각으로 미리 걸어두는" 장시간(최대 12시간) 용도. 전투/자동사냥과 전혀 안 겹침(몬스터를 "떼어가는" 게 아니라 그냥 병행되는 타이머일 뿐이라 앱을 켜놓고 계속 플레이해도 무방).
- **골드 공식을 `calc_idle_gold`(2.5초 틱 기준)를 재사용하지 않고 완전히 새로 만듦**: 12시간을 틱 단위로 환산하면 `add_gold`의 100만 상한을 손쉽게 넘어 크래시하는 걸 시뮬레이션으로 미리 확인함(117/118에서 겪은 것과 같은 클래스의 함정을 사전에 피함). 대신 `calc_expedition_gold_per_hour(chapter, level)` — `calc_idle_gold`와 같은 v_hp 베이스를 쓰되 배율을 15로 훨씬 작게 잡은(calc_idle_gold는 2.5초당 40배) "시간당 요율" 전용 공식을 신설. 최종 결과는 그래도 방어적으로 100만 클램프.
- **파견 슬롯 다중화 (migration 132, 신규, 사용자 요청)**: 기존엔 유저당 1개만 동시에 보낼 수 있었는데, **레벨 100마다 슬롯이 1개씩 늘어남**(`calc_expedition_slots(level) = 1 + floor(level/100)` — 레벨100=2개, 200=3개, ..., 만렙500=6개). `expeditions` 테이블을 `user_id` 단독 PK에서 별도 `id`(uuid) PK로 바꿔서 유저당 여러 행을 가질 수 있게 함. `start_expedition`/`claim_expedition` 둘 다 반환/인자가 바뀌어(`claim_expedition`이 이제 `p_expedition_id`를 받음) DROP FUNCTION 포함. 조회용 `fetch_my_expeditions()`가 진행 중인 파견 전부 + 총 슬롯 수를 같이 반환. 클라이언트/서버 둘 다 슬롯 계산식을 각자 갖고 있으므로(`expedition.js`의 `calcExpeditionSlots`) **하나 바꾸면 반드시 둘 다 같이 바꿔야 함**.
- 클라이언트(`ExpeditionPanel`, `DungeonSelect.jsx` 내부)는 1초마다 남은시간을 재계산해서 표시, 완료되면 "수령" 버튼으로 전환. 슬롯이 남아있을 때만 새 파견 시작 버튼 노출.
