## 뽑기 스킬 카탈로그 (50종)

`skillCatalog.js`의 `SKILL_CATALOG` — 등급당 **10종 × 5등급 = 총 50종**(006 시드 15종 + 017 추가 35종). ⚠️ 서버 `skill_catalog`와 값이 예전엔 "반드시 동일해야 함"이었는데, 등급별 격차를 크게 벌리며(아래) 재확인해보니 서버 `base_value` 컬럼은 어떤 함수도 실제로 select해서 쓰지 않는 **순수 시드/장식용 컬럼**이었음(전투 데미지 계산이 전부 클라이언트에서 일어남, [`security.md`](./security.md)의 "알려진 한계" 참고) — 그래서 이번 변경은 클라이언트 값만 바꾸고 서버는 의도적으로 그대로 둠(마이그레이션 불필요).

등급별 구성: 4종 damage / 2종 heal / 1종 stun / 1종 dot / 1종 buff(atk 또는 def) / 1종 haste

- **등급별 격차 대폭 확대 (신규, 순수 클라이언트 변경, 사용자 요청)**: 기존엔 등급마다 base가 균일하게 ×1.4씩만 벌어져 있어서(노멀1.0→신화3.84, 총 3.84배), 스킬레벨 성장폭(레벨1000 기준 최대 ×3.997)이 등급 간 격차보다 더 커서 **"낮은 등급을 만렙까지 키운 게 높은 등급 초반보다 세다"는 역전 현상**이 있었음(사용자 피드백: "등급별로 비등비등한 것 같아").
  - **damage 타입**은 등급마다 **×2.5(레어)/×6.5(에픽)/×16(전설)/×40(신화)** — 노멀 대비 최대 **40배**까지 압도적으로 벌림(정확히 "압도적으로" 요청 반영)
  - **heal/stun/dot/buff/haste**는 회복%·기절초·쿨감% 같은 절대적 의미가 있는 수치라 damage와 같은 배율을 쓰면 회복 1800%·기절 100초처럼 게임을 깨뜨리는 값이 나옴 — 대신 **×1.6/×2.6/×4.2/×6.8**로 훨씬 완만하게(그래도 기존 ×1.4/1.96/2.744/3.84보다는 전부 큼, 하향 없음) 적용
  - **보유효과(`getSkillPossessionBonus`)는 완전히 별도 테이블(`SKILL_POSSESSION_BASE`)을 쓰고 있어 이번 변경과 무관 — 그대로 유지됨**(사용자 요청 그대로 반영)
  - haste 타입은 이미 `Math.min(0.95, ...)` 안전 클램프가 있어(BattleScreen.jsx, 유물 버프% 연동 시 도입) 큰 배율이 들어와도 쿨타임이 0 이하로 내려가지 않음 — heal은 `Math.min(hp+heal, maxHp)`로 이미 최대체력 초과분이 자연히 무시되는 구조라 안전
- **스킬레벨 성장식**: `실효값 = base × (1 + (level-1)×0.003)` — 원래 레벨100 상한일 때는 최대 성장폭이 약 ×1.3에 불과해, 등급 간 base 차이(×1.4)를 아무리 강화해도 못 넘도록 의도했었음. **039에서 레벨 상한을 1000으로 올리며 수식은 유지**했다가 성장폭이 등급 격차를 역전하는 문제가 생겼는데, 위 등급별 격차 확대로 근본적으로 해결됨
- **중복 뽑기 시 새 스킬 대신 기존 `skill_level`이 +3 상승**(최대 **1000**, 039에서 100→1000 상향) — `user_skills`의 `unique(user_id, skill_key)` 제약으로 중복 row는 안 생김

## 스킬 타입별 전투 로직

`BattleScreen`/`DungeonBattle`/`JobDungeonBattle` 세 화면에 동일 구현. 값 해석 기준은 `skillCatalog.js` 상단 주석 참고:

- **`damage`**: `mitigateDamage(atk*배율, 상대방어력)`
- **`heal`**: 최대체력 대비 %
- **`stun`**: `base` = 기절 지속시간(초). 적 자동공격 인터벌이 매 틱 `Date.now() < enemyStunnedUntil`을 체크해 기절 중이면 공격 스킵
- **`dot`**: `base` = 틱당 데미지 배율, `ticks`/`tickInterval`(카탈로그 고정값)만큼 `setTimeout` 예약. 시전 시점 방어력 기준으로 틱뎀 확정(성장/버프 변동 미반영, 단순화)
- **`buff_atk`/`buff_def`**: `base` = 스탯 증가 배율, `duration`(고정)만큼 지속. `playerBuffs` state에 `{atkUntil,atkMult}`/`{defUntil,defMult}`로 저장, 매 순간 `Date.now()`와 비교해 활성 여부 판정
- **`haste`**: `base` = 쿨타임 감소 비율, `duration`(10초, 8초→10초 상향)만큼 지속. 즉시 발동 중인 다른 스킬 타이머를 되돌리진 않고, **헤이스트가 켜진 동안 스킬을 쓸 때마다 그 스킬의 쿨타임 자체가 줄어서 적용**됨
  - ⚠️ **버그 수정(사용자 제보)**: 헤이스트가 실제로 끝나도 "⚡ 쿨타임 감소 중" 아이콘이 안 사라지던 문제. 원인은 표시 조건 `Date.now() < playerBuffs.hasteUntil`인데 **React는 시간이 흘렀다고 저절로 리렌더 안 함**(효과 자체는 다음 스킬 사용 시점에 정확히 사라지고 있었음, 순수 UI 지연). 헤이스트 걸 때 `setTimeout(..., skill.duration)`으로 버프 종료 시점에 강제 리렌더를 예약해 수정. 5개 전투 화면(PvP연출 제외) 전부 적용

부가 UI:
- 버프/기절 상태는 `BuffStatusRow`가 HP바 아래 배지로 표시(⚔️공격력상승/🛡️방어력상승/💫적기절중). 헤이스트는 배지 대신 캐릭터 위 아이콘으로만 표시(중복 정보 정리)
- 헤이스트로 쿨타임이 줄면 `SkillButton`에 `skill.cooldown` 대신 `effectiveCooldowns[skill.id]`를 넘겨 링 애니메이션도 정확히 맞춰 돎
- **캐릭터 위 상태 아이콘**: 헤이스트(⚡, 지속시간 동안 회전)와 회복(💚, 1.3초 팝업)은 배지뿐 아니라 **플레이어 스프라이트 바로 위**에도 표시(`player-status-fx`, 4개 전투화면 전부)
- **전직(각성) 스킬 전용 강화 이펙트**: `getJobSkillTier(skillId)`로 전직 스킬 여부 판별, 맞으면 **차수가 높을수록 파티클이 커지고**(`spawnParticles`의 `sizeMult`), **3차 이상부터 타격 순간 등급별 색상 화면 플래시**(`job-skill-flash`, `mix-blend-mode: screen`). 1~5차 색상은 노랑→주황→빨강→핑크→보라

## 스킬 편성

- **뽑기 화면과 완전히 분리된 최상위 탭**(하단 게임 탭 "🧩 스킬편성", `SkillLoadout.jsx`) — 041에서 스테이지 탭 옆 최상위 탭으로 승격
- 카드 색상: **금색 테두리+"착용중" 배지**는 서버에 실제 저장된 스킬, **청록색 테두리**는 편성 화면에서 선택만 하고 "편성 저장"을 안 누른 상태(저장 전까지 실제 장착에 영향 없음)
- 편성 슬롯+저장 버튼이 **`position: sticky`로 상단 고정**(`loadout-sticky-bar`) — 헤더도 이미 sticky라 `top`을 "헤더 높이+15px"로 잡음(데스크톱 84px, 모바일 76px)
- 편성바 상단 토글(`▲ 접기` / `▼ 펼치기`)로 슬롯 영역 전체를 접었다 펼 수 있음
- 장착 슬롯 수는 활성 몬스터 레벨에 따라 1→**10**개(`getSkillSlotCount`: Lv10/25/50/75/100/130/160/190/220, 041에서 5→10 상향, 서버 동기화)
- `profiles.equipped_skills`(text[])에 저장, `set_skill_loadout`이 슬롯수/보유여부/중복을 서버 재검증
- `BattleScreen`은 고정 스킬 대신 **App.jsx가 계산한 `equippedSkills` prop**을 사용. 장착 스킬 0개인 예외 상황엔 `skills.js` 첫 스킬로 폴백. 전직 스킬은 `getAvailableSkills`가 추가로 붙임

## 전직 스킬 전용 키보드 단축키 (신규, 사용자 요청)

기존 숫자키(1~9) 단축키는 `availableSkills` 배열의 **순번**대로 매핑돼있어서, 기본 스킬(최대 5개) + 전직 스킬(최대 10개, 127에서 6~10차 확장)을 다 채우면 전직 스킬 상당수가 9번째 순번을 넘어가 **단축키가 아예 없는 상태**였음(사용자 피드백: "전직스킬을 마우스로만 사용하기 너무 힘들다").

- `lib/keybinds.js`가 전직 **단계(1~10차)**별로 고정된 전용 키를 배정 — 기본값 `Q W E R A S D F Z X`(+여유 2개 `C V`, 총 12슬롯) — 장착한 스킬이 몇 개든, 몇 번째 순번이든 **항상 같은 키로 그 단계 전직스킬을 사용**할 수 있음
- 4개 전투화면(`BattleScreen`/`DungeonBattle`/`JobDungeonBattle`/`WorldBossBattle`) 전부 동일하게 적용 — 각 화면의 키다운 핸들러에 `getJobSkillTier(skill.id)`로 전직스킬 여부/단계를 판별해 배정된 키와 매치
- `SkillButton`의 hotkey 뱃지도 전직스킬이면 숫자 대신 배정된 문자를 표시(대문자로)
- ⚠️ **BattleScreen의 R키 충돌 주의**: 기본값에서 4차 전직 키가 `R`인데, 같은 화면에 이미 "패배 후 R로 재도전" 단축키가 있었음. 실제로는 겹치지 않음 — 전직스킬 R 처리는 `mode==='challenge' && !result`(전투가 진행 중일 때만) 조건으로 이벤트를 소비하고, 전투가 끝나 `result`가 채워지면 그 조건이 꺼져서 자연스럽게 재도전 핸들러로 넘어감. 다만 **커스텀 키 설정에서 다른 키 조합이 우연히 겹칠 가능성은 검증하지 않음**(사용자가 직접 고르는 값이라 전부 사전 차단하기 어려움) — 후속 검토 항목

## 설정 > 키보드 구성 (신규, 사용자 요청, `KeybindSettings.jsx`)

설정 화면에 "⌨️ 키보드 구성" 탭 추가 — 슬롯을 클릭하고 원하는 키를 누르면 바로 재배정. 이미 다른 슬롯이 쓰던 키를 고르면 그 슬롯은 자동으로 비워짐(중복 방지, 자리 맞바꿈 대신 명시적으로 "비어있음"을 보여줘서 저장 전 사용자가 직접 채우게 유도). `localStorage`에 저장(서버 동기화 없음 — 순수 UX 취향이라 기기별로 달라도 무방).

- **모바일엔 노출 안 함**(사용자 요청 — "모바일에선 필요없으니") — `.desktop-only-tab` 클래스로 기존 640px 브레이크포인트에서 `display: none` 처리. 탭 버튼 자체가 안 보이니 모바일에서 이 화면으로 진입할 방법이 없음(별도 라우트 가드 불필요)

## 뽑기 비용/레벨

- 뽑기 비용은 **레벨 연동 동적가**: `300 + (뽑기레벨-1)×90`(Lv.1=300골드~Lv.50=4,710골드). 015에서 고정가 300으로 바뀌었다가 **045에서 다시 동적가로**(장비뽑기와 동일 패턴 통일)
- **뽑기 레벨**: `1 + floor(누적뽑기횟수/1000)`, 최대 **50**(045에서 20→50 확장). 레벨 구간별로 고등급 확률이 계단식 상승(구간 경계 8/18/28/38/48)
- **1/10/100연차 뽑기** 지원(`draw_skill_batch`) — 골드 부족하면 그 시점까지만 부분 성공
- 신규 유저는 스타터 생성 시 `basic_strike` 1개를 무료 자동 지급+장착(`create_starter_monster`)
- `G`/`Shift+G`/`Ctrl+G` 단축키로 1/10/100회 뽑기([`ui-and-ux.md`](./ui-and-ux.md))
- ⚠️ **버그 수정**: 뽑을 때마다 서버는 `total_skill_draws`를 증가시키는데, **클라이언트가 뽑은 뒤 그 값을 다시 안 불러와서** 뽑기레벨/진행바가 새로고침 전까지 멈춰있던 버그(장비뽑기는 이미 고쳐져 있었는데 스킬뽑기의 `refreshSkills`는 `profile`을 빠뜨렸던 게 원인). `refreshSkills`가 `fetchUserSkills`와 `getMyProfile`을 함께 호출하도록 수정
- **중복(강화) 시 10% 확률 럭키 보너스**(055) — 평소 +3인 강화량이 10% 확률로 +6(2배). 장비도 동일 원리([`equipment.md`](./equipment.md)). 반환 컬럼 그대로라 클라이언트 수정 없이 결과 숫자가 크게 찍히는 것으로만 체감
- **뽑기 확률 투명 공개**(신규, `lib/gachaProbability.js`) — "🎲 현재 확률 보기" 토글로 뽑기레벨 기준 정확한 확률(%)을 표시. 서버 확률표(055)를 클라이언트에 그대로 미러링, diff로 6개 구간 수치 일치 확인. **⚠️ 서버 확률을 바꿀 땐 `gachaProbability.js`도 반드시 함께 갱신할 것**(안 그러면 화면 확률이 실제와 어긋남) — 이 규칙을 파일 상단 주석에도 명시

## 일일 무료 뽑기 (migration 049, 101에서 5종 분리)

상점 상단 "🎁 오늘의 무료뽑기"로 **하루 1회 골드 소비 없이** 현재 탭을 뽑을 수 있음. 출석체크와 별개 장치("가챠를 무료로 한 번 더" — 가챠 장르의 흔한 재방문 유도).

**구현 방식**: 별도 확률표 복제 없이 **기존 `draw_skill()`/`draw_equipment(slot)`을 그대로 호출**해 단일 확률 로직 유지. 정상적으로 골드가 차감되므로, `claim_daily_free_draw`가 호출 직전 잔액을 기억해뒀다가 뽑기 후 그 값으로 강제 복원해 "무료"를 보장. 잔액 부족 실패를 막기 위해 뽑기 직전 임시 버퍼(10만 골드)를 잠깐 지급했다가 마지막에 정확히 원래 잔액으로 리셋 — `security definer` 안에서 본인 잔액만 건드리므로 안전하고, 확률표가 바뀌어도 무료뽑기는 따로 수정할 필요 없음.

무료로 뽑아도 뽑기레벨/업적(`gacha_100` 등)에 그대로 반영됨.

### 5종(무기/방어구/장갑/신발/스킬) 독립 분리 (migration 101, 사용자 요청)

원래 "스킬 또는 장비 중 1종만 하루 1회"였는데, 이제 **5종 전부 각각 독립적으로 하루 1회씩** 무료. `daily_free_draw_state`를 `(user_id, draw_type)` 복합키로 재구성. `claim_daily_free_draw(p_type)`의 `p_type`이 슬롯/스킬 구분값을 겸하도록 단순화, `p_slot` 파라미터 제거(시그니처 변경이라 DROP FUNCTION 선행).

- 정책이 "하루 1개"→"하루 5개"로 바뀌는 시점이라 기존 상태 테이블을 통째로 재생성 — 배포 직후 유저 전원이 5종을 새로 받게 되는 정도라 경제 영향 미미
- 클라이언트도 단일 boolean → 5종 map 구조(`buildFreeDrawUsedMap`)로 재작성, 5개 탭 버튼 각각에 "무료뽑기 가능" 점 표시
- 헤드리스 시뮬레이션으로 부분사용/신규유저/조회실패 세 케이스 검증

## 스킬 보유효과

`getSkillPossessionBonus`/`sumSkillPossessionBonus`(`skillCatalog.js`) — 장비처럼 스킬도 **보유만 하면(장착 무관) 상시 공격력 보너스**.

- 등급별 기준치: 노멀 2 / 레어 4 / 에픽 8 / 전설 16 / 신화 32
- `1 + (스킬레벨-1)*0.01`을 곱해 레벨이 오를수록 커짐
- `App.jsx`가 `sumSkillPossessionBonus(userSkills)`로 합산해 장비 보너스에 더함 — 최종 전투력엔 "장착 보너스 + 장비 보유효과 + 스킬 보유효과"가 전부 누적
- 스킬 뽑기 화면에 개별 카드 및 전체 합계 표시

## 스킬 완전수집 업적 (migration 092, 신규 콘텐츠)

"만능 마스터" — 50종 전부 보유 시 달성. `user_skills`의 `(user_id, skill_key)` unique 제약으로 `count(distinct skill_key)` 정확 판정. 코스튬 완전수집(077)과 동일한 엔드게임 목표. CASE만 추가, DROP 불필요, diff/42개 키 재검증 완료.
