# PvP

관련 파일: `PvP.jsx`, `PvPArena.jsx`, `PvPBattleScene.jsx`, `PvPShop.jsx`, `pvp.js`, migration 023/024/025

## 개요

- **비동기 매칭 전투** — 실시간 대결이 아니라 도전 버튼을 누르면 서버가 즉시 상대를 찾고 승패까지 한번에 판정
- **결과를 즉시 보여주지 않고**, `PvPBattleScene.jsx`가 이미 정해진 결과(승/패, 양측 전투력)를 약 2.7초(6라운드×450ms) HP바 애니메이션으로 먼저 연출 후 결과 패널 표시(`PvPArena.jsx`가 `pendingBattle`/`lastResult` 두 단계로 분리 관리)
- "이미 정해진 결과를 재생"하는 방식이지만 타격감을 위해 다른 전투화면과 동일한 캔버스 파티클을 붙임: 매 라운드 양쪽에서 파티클(이기는 쪽이 더 화려)이 터지고, 라운드마다 흰색 `job-skill-flash`(각성 스킬과 동일 컴포넌트, [`skills.md`](./skills.md))가 짧게 뜨고, 마지막 라운드엔 승/패 색상(금색/빨강)으로 크고 오래 번쩍임. 매 라운드 랜덤 타격 로그(`HIT_LOGS`)도 표시
- 화면 흔들림은 `pvp-battle-scene.shake` 전용 규칙으로 일반 전투화면보다 강하게
- **투사체**: 매 라운드 양쪽에서 서로에게 광구(꼬리+머리, 포물선 궤적)가 날아다님(`spawnProjectile`) — 이기는 쪽 3발, 지는 쪽 1발로 우세를 시각화, 도달 시 작은 폭발 파티클. 마지막 라운드엔 승자 쪽 5발 집중포화

## 매칭 로직

`start_pvp_battle` RPC:

1. 활성 몬스터가 있는 다른 유저 중 무작위 1명
2. 그 사람 전투력이 **내 전투력의 75~125%** 안이면 그대로 매칭
3. 범위 밖이거나 대상이 없으면 **내 전투력 기준 90~110% 랜덤인 가상 캐릭터**로 대체(고정 7개 이름 후보 중 랜덤)

## 전투력 계산 (PvP 전용, 서버)

`calc_monster_stats()`가 종족 base + 레벨 성장 + 전직 배율(1~5차)로 "만피 기준" atk/def/maxHp를 재계산, `calc_combat_power()` = `atk*4.5+def*3.2+maxHp*0.6`(클라이언트 공식과 동일).

**장착 장비 보너스 반영 (051)**: `calc_equipped_stat_bonus(user_id)`가 장착 중인 4슬롯을 조회해 `itemCatalog.js`와 동일 공식(슬롯 base × 등급 배율 × (1+강화수치×0.08))으로 보너스를 계산, `calc_monster_stats()` 결과에 더해 `calc_combat_power()`에 넣음. **장비 강화가 실제 체감을 압도적으로 좌우하는 게임이라**, 원래 종족+레벨+전직만 보던 방식은 강화 안 한 유저와 신화 풀강 유저를 똑같이 취급하는 불공정이었음 — 051에서 수정. `start_pvp_battle`/`fetch_leaderboard`/`fetch_my_rank`/`fetch_my_combat_power` 전부 동일 기준으로 통일됨.

**스킬 보유효과는 여전히 미반영** — 50종을 SQL로 포팅하려면 계산식 전체를 옮겨야 해 장비보다 범위가 큼([`todo.md`](./todo.md)).

⚠️ `calc_equipped_stat_bonus`는 `itemCatalog.js`의 `SLOTS.base`/`RARITIES.statMultiplier`를 SQL에 그대로 하드코딩 포팅한 것이라, **클라이언트 장비 밸런스 수치가 바뀌면 이 SQL 함수도 반드시 같이 고쳐야 함**(안 그러면 PvP/랭킹 전투력과 인벤토리 화면 스탯이 어긋남).

## 승패 판정

양측 전투력에 각각 **±15% 랜덤 보정**을 곱해서 비교, 높은 쪽 승리(약간의 운 요소). 정교한 턴제 시뮬레이션은 안 함(단순화) — 애니메이션은 이미 정해진 결과를 재생할 뿐.

## PvP 재화/전적

- 승리 시 `reward = max(20, 20 + 상대전투력/65)` 획득(`pvp_currency`, 025에서 이전보다 약 30% 하향), 패배 시 없음
- **실제 유저 상대 보상 강화 (migration 107, 사용자 요청)**: 상대가 가상 캐릭터가 아니라 실제 유저(`opponent_is_real=true`)면 승리 보상이 **3배**로, 패배해도 원래 승리 보상(1배)만큼 **위로 보상**이 지급됨. 가상 캐릭터 상대는 기존 그대로(승리만 1배, 패배 0). 매칭 실패 시 자동으로 가상 캐릭터가 나오는 구조라, "리스크 없는 가상 상대만 계속 도전"보다 "실제 사람과 붙는 게 확실히 이득"이 되도록 유도하는 목적 — 소셜 대전의 재미를 자연스럽게 끌어올리는 인센티브 설계
- 전적은 `pvp_wins`/`pvp_losses`에 누적, `pvp_battle_log`에 매 전투 기록. `fetchPvpHistory()`가 구현돼있었는데 실제로는 어디서도 호출 안 되는 죽은 코드였음 — `PvPArena.jsx`에 "▼ 최근 전적 보기" 토글로 연결해서 최근 20전(승/패, 상대, 전투력, 재화)을 볼 수 있게 함. 목록이 열려있으면 전투 직후 자동 최신화
- **연승 스트릭 표시**: 최근 전적을 앞에서부터 훑어 'win' 연속 개수를 클라이언트에서 계산, 2연승 이상이면 "🔥 N연승 중!" 배지. 서버 추가 없이 이미 로드한 데이터로만 계산(전투 종료마다 재조회하도록 바꿔 즉시 반영)

## 복수전(리벤지) (migration 108, 신규 콘텐츠)

107로 실유저 대전 보상이 크게 강화되면서 자연스럽게 생기는 "그 상대 다시 붙어보고 싶다"는 동기를 실제 기능으로 연결. 최근 전적 목록에서 실유저 상대 행마다 "🔁 복수하기" 버튼이 뜨고, 누르면 무작위 매칭 대신 **그 상대를 지목해서** 바로 재대결.

- `start_pvp_revenge_battle(p_opponent_id)` — `start_pvp_battle`과 거의 동일한 로직이지만 매칭 단계를 건너뛰고 상대를 고정. 상대의 스탯은 **매번 실시간으로 다시 조회**(옛 전적에 박제된 스탯이 아니라 상대가 그 사이 강해졌으면 그만큼 반영)
- 상대는 항상 실제 유저이므로 107의 "승 3배/패도 위로보상" 규칙이 **무조건 적용**됨
- 상대가 그 사이 몬스터를 바꿨거나(활성 몬스터 없음) 등으로 못 찾으면 에러 반환, 클라이언트가 토스트로 안내
- 자기 자신을 지목하는 것은 서버에서 차단(`p_opponent_id = auth.uid()`)
- 쿨다운(`last_pvp_battle_at` 2초)과 반환 컬럼 구성은 `start_pvp_battle`과 동일하게 공유 — 신규 함수라 DROP FUNCTION 불필요
- **당한 쪽에게도 알림 (migration 109)**: 복수전이 끝나면 당한 상대(원래 로그의 주인)에게도 결과+소액 위로 골드(`greatest(10, base_reward*0.3)`)를 우편으로 발송. "누가 나한테 복수전을 걸었는지" 알아야 재접속해서 되갚아주고 싶은 동기가 생기므로 — 순수 파밍 방지 목적의 107/108과 달리 **재접속 유도** 목적의 장치. `source_key`에 `gen_random_uuid()`를 섞어 매번 새 우편으로 취급(같은 상대에게 여러 번 복수전을 걸어도 전부 개별 우편으로 쌓임)
- ⚠️ **알려진 밸런스 특성**: `start_pvp_battle`의 무작위 매칭은 상대 전투력이 내 ±25% 밖이면 가상 캐릭터로 대체하지만, `start_pvp_revenge_battle`은 **이 범위 제한이 없음** — 최근 전적에 뜬 상대라면 전투력 격차가 아무리 크더라도 지목 가능. 강한 유저가 약한 유저를 반복 지목해 사실상 확정승을 파밍할 여지가 있으나, 107의 "패배해도 위로보상" 덕분에 당하는 쪽도 완전히 빈손은 아니고, 2초 쿨다운(`last_pvp_battle_at` 공유)이 반복 속도 자체는 제한함. 실제 플레이 반응 보고 더 심한 어뷰징 신호가 보이면 전투력 격차 제한 추가를 검토(사용자 확인 없이 지금 단계에서 손대지 않음, [`todo.md`](./todo.md))
- **복수전 승리 업적 (migration 113, 신규 콘텐츠)**: `profiles.pvp_revenge_wins`(일반 PvP 승리와 별개, 복수전 승리만 카운트)가 10 이상이면 "🔁 복수의 화신" 업적(8000골드) 수령 가능. `claim_achievement`/`ACHIEVEMENT_CATALOG` 둘 다 pvp 카테고리에 추가, 둘 다 반환타입 그대로라 DROP FUNCTION 불필요
- **결과 공유 버튼 (신규 콘텐츠)**: 실유저 상대 승리 시에만 결과카드에 "📋 결과 공유" 버튼 노출(가상 상대/패배는 굳이 자랑할 거리가 아니라 제외) — 뽑기 결과 공유([`equipment.md`](./equipment.md))와 동일한 클립보드 복사 패턴 재사용
- ⚠️ **[예방적 버그 수정] 보상 오버플로우 (migration 117)**: `add_gold`는 amount가 100만을 넘으면 예외를 던지는데(030), 상대 전투력이 약 2165만 이상이면 `base_reward*3`(실유저 승리 보너스)이 100만을 넘을 수 있음. 레벨 상한이 없는 게임이라 장기적으로 이 구간에 도달하는 유저가 나올 수 있어, 실제로 터지기 전에 `start_pvp_battle`/`start_pvp_revenge_battle` 둘 다 `add_gold` 호출 직전 `v_reward`를 90만으로 클램프. 110(월드보스 크래시)을 계기로 유사 리스크를 미리 점검하다 발견

## PvP 티어 시스템 (신규 콘텐츠, `lib/pvpTier.js`)

`pvp_wins` 값을 구간별로 나눠 브론즈(0)/실버(10)/골드(30)/플래티넘(60)/다이아몬드(100) 5단계 명예 등급 부여. **스탯 보너스 없는 순수 표시용**(밸런스 영향 없음) — 승수 자체가 이미 서버가 관리하는 신뢰 값이라 순수 클라이언트 계산.

- 전투력 카드 최상단에 현재 티어 배지 + "다음 티어까지 N승" 표시
- 극단값(억 단위 전투력, 100승 이상)에서도 모바일 카드 넘침 없음을 재검증

- ⚠️ **보안**: 미션 클레임과 동일하게 `last_pvp_battle_at` 기준 **최소 2초 쿨다운**(원래 20초, 038에서 단축, 무한 반복 파밍 방지 목적 유지, [`security.md`](./security.md))

### 버그 수정 이력

전투 직후 "🏆 N승 · 💀 M패" 표시가 **다음 전체 프로필 재조회 전까지 안 바뀌는** 버그 — `PvPArena.jsx`가 재화(`pvp_currency`)만 상위로 올려주고 승패는 반영 경로가 없었던 게 원인. `onCurrencyChange`를 `onBattleResolved(res)`로 바꿔서 `App.jsx`가 `res.result`로 `pvp_wins`/`pvp_losses`까지 즉시 갱신하도록 수정.

## PvP 상점 (코스튬)

- 전체 유저 공용 진열대 **10칸**(031에서 30칸→10칸 축소), **매시 정각 자동 갱신**(`sync_pvp_shop`, 우편함과 동일한 lazy-generation — cron 없이 이번 시간대 진열대가 없으면 그때 생성)
- **다음 갱신까지 실시간 카운트다운**(`useCountdownToNextHour`) — 로컬시각 기준 "다음 정시"까지 MM:SS로 1초마다 갱신. 서버(`date_trunc('hour', now())`, 보통 UTC)와 한국(UTC+9, 정수 오프셋)은 정시가 항상 같은 순간이라 로컬 계산이 정확히 일치함. 0에 닿으면 서버가 만들 시간을 1.2초 준 뒤 자동 재조회
- 무기/방어구/장갑/신발 × 5등급 조합을 등급 가중치로 랜덤 추첨 — 노멀45% / 레어27% / 에픽16% / 전설8% / 신화4%(031에서 재조정, 이전 40/28/18/10/4)
- 가격은 등급별 기준가에 ±20% 랜덤 가산:

| 등급 | 기준가 |
|---|---|
| 노멀 | 3,000 |
| 레어 | 8,000 |
| 에픽 | 20,000 |
| 전설 | 55,000 |
| 신화 | 150,000 |

- 승리 1회 평균 보상이 수십~백여 골드라, 노멀 코스튬 하나 사려면 최소 수십 회 승리 필요 — "노멀도 꽤 비싸다"는 요구사항 반영

### 버그 수정 이력 — 지난 시간대 진열대가 계속 쌓여서 보이던 문제

`sync_pvp_shop`이 매시간 새 진열대(`period_key`)를 **추가만 하고 예전 걸 안 지웠고**, 클라이언트도 필터링 없이 테이블 전체를 조회해서 시간이 지날수록 여러 시간대가 뒤섞여 보이던 문제(30개 시절엔 더 심함).

→ **수정**(032): `sync_pvp_shop`이 새 진열대를 만들기 직전 `delete from pvp_shop_listings where period_key <> 이번시간`으로 정리(근본 해결). 클라이언트도 이중 방어로 최신 `period_key`만 걸러 조회.

### 코스튬의 성격

**전투 스탯에 영향 없는 수집/과시용** — `pvp_costume_inventory`에 보유 여부 기록, 기존 장비 인벤토리(`user_inventory`)와 완전히 분리된 별도 테이블.

### 코스튬 착용 시스템 (migration 042)

기존엔 "보유"만 있고 "착용" 개념 자체가 DB에 없었음. 042에서 추가:

- `profiles.equipped_costumes`(text[], `equipped_skills`와 동일 패턴) — 슬롯당 최대 1개, `item_key`가 `슬롯_등급` 형태라 접두어로 슬롯 판별
