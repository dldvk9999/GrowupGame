# PvP

관련 파일: `PvP.jsx`, `PvPArena.jsx`, `PvPBattleScene.jsx`, `PvPShop.jsx`, `pvp.js`, migration 023/024/025

## 개요

- **비동기 매칭 전투** — 실시간 대결이 아니라, 도전 버튼을 누르면 서버가 그 즉시 상대를 찾고 승패까지 한번에 판정함
- **결과를 즉시 보여주지 않고**, `PvPBattleScene.jsx`가 서버가 이미 정한 결과(승/패, 내 전투력, 상대 전투력)를 약 2.7초짜리(6라운드×450ms) HP바 애니메이션으로 먼저 연출한 뒤에 결과 패널을 표시함(`PvPArena.jsx`가 `pendingBattle`/`lastResult` 두 단계 state로 분리 관리)
- 진짜 실시간 조작 대전은 아니고 "이미 정해진 결과를 그럴듯하게 재생"하는 방식

## 매칭 로직

`start_pvp_battle` RPC:

1. 활성 몬스터가 있는 다른 유저 중 무작위 1명을 뽑음
2. 그 사람 전투력이 **내 전투력의 75~125% 범위** 안이면 그대로 매칭
3. 범위 밖이거나 매칭 대상이 아예 없으면 **내 전투력 기준 90~110% 랜덤인 가상 캐릭터**를 대신 세움 (이름은 "그림자 전사" 등 정해진 7개 후보 중 랜덤)

## 전투력 계산 (PvP 전용, 서버)

`calc_monster_stats()`가 종족 base 스탯 + 레벨 성장 + 전직 배율(1~5차)로 "만피 기준" atk/def/maxHp를 다시 계산하고, `calc_combat_power()` = `atk*4.5+def*3.2+maxHp*0.6`(클라이언트 `combat.js`의 공식과 동일하게 맞춤).

⚠️ **장비/스킬 보유효과는 PvP 전투력 계산에 포함되지 않음** — 서버에서 매번 인벤토리를 조인해서 계산하는 부담을 피하려는 의도적 단순화. 장비/스킬 비중이 커지면 나중에 반영 검토 필요(자세한 내용은 [`todo.md`](./todo.md)).

## 승패 판정

양측 전투력에 각각 **±15% 랜덤 보정**을 곱해서 비교, 높은 쪽이 승리(약간의 운 요소). 정교한 턴제 시뮬레이션은 하지 않음(단순화) — 애니메이션(`PvPBattleScene.jsx`)은 이미 정해진 결과를 시각적으로만 재생하는 것.

## PvP 재화/전적

- 승리 시 `reward = max(20, 20 + 상대전투력/65)`을 획득(`profiles.pvp_currency`, migration 025에서 기존 `max(30, 30+상대전투력/50)`보다 약 30% 하향), 패배 시 없음
- 전적은 `profiles.pvp_wins`/`pvp_losses`에 누적, `pvp_battle_log`에 매 전투 기록 남음
- ⚠️ **보안**: 미션 클레임과 동일하게 `profiles.last_pvp_battle_at` 기준 **최소 20초 쿨다운**을 서버에 걸어둠(무한 반복 파밍 방지, 자세한 내용은 [`security.md`](./security.md))

### 버그 수정 이력

전투 직후 화면의 "🏆 N승 · 💀 M패" 표시가 **다음 전체 프로필 재조회 전까지 안 바뀌는** 버그가 있었음 — `PvPArena.jsx`가 재화(`pvp_currency`)만 상위로 올려주고 승패 전적은 상위 `profile` state에 반영하는 경로가 아예 없었던 게 원인. `onCurrencyChange` 대신 `onBattleResolved(res)` 콜백으로 바꿔서, `App.jsx`가 `res.result`를 보고 `pvp_wins`/`pvp_losses`까지 함께 즉시 갱신하도록 수정함.

## PvP 상점 (코스튬)

- 전체 유저 공용 진열대 **10칸**(migration 031에서 30칸→10칸으로 축소), **매시 정각마다 자동 갱신**(`sync_pvp_shop` RPC, 우편함과 동일한 lazy-generation 패턴 — cron 없이 "이번 시간대 진열대가 아직 없으면 그때 생성")
- 무기/방어구/장갑/신발 × 5등급 조합 중 등급 가중치로 랜덤 추첨 — **등급이 높을수록 뚜렷하게 희귀함**: 노멀45% / 레어27% / 에픽16% / 전설8% / 신화4%(migration 031에서 재조정, 이전엔 40/28/18/10/4)
- 가격은 등급별 기준가에 ±20% 랜덤 가산:

| 등급 | 기준가 |
|---|---|
| 노멀 | 3,000 |
| 레어 | 8,000 |
| 에픽 | 20,000 |
| 전설 | 55,000 |
| 신화 | 150,000 |

- 가격 설계 의도대로면 승리 1회 평균 보상이 대략 수십~백여 골드 수준이라, 노멀 코스튬 하나 사려면 최소 수십 회 승리가 필요함 — "노멀도 꽤 비싸다"는 요구사항을 반영한 수치

### 버그 수정 이력 — 지난 시간대 진열대가 계속 쌓여서 보이던 문제

`sync_pvp_shop`이 매시간 새 진열대(`period_key`)를 **추가만 하고 예전 것을 지우지 않았고**, 클라이언트(`fetchPvpShop`)도 `period_key`로 필터링 없이 `pvp_shop_listings` 테이블 전체를 조회해서 보여주고 있었음. 그래서 시간이 지날수록 "이번 시간 10개 + 지난 시간 10개 + 그 전 시간 10개 ..."가 전부 섞여서 표시되는 문제가 있었음(30개 시절엔 더 심했음).

→ **수정**(migration 032): `sync_pvp_shop`이 새 시간대 진열대를 만들기 직전에 `delete from pvp_shop_listings where period_key <> 이번시간`으로 예전 시간대를 정리하도록 변경(근본 원인 해결). 클라이언트도 이중 방어로 `fetchPvpShop`이 가장 최신 `period_key`만 걸러서 조회하도록 수정.

### 코스튬의 성격

**코스튬은 전투 스탯에 영향 없는 수집/과시용**임 — `pvp_costume_inventory`에 보유 여부만 기록되고, 기존 장비 인벤토리(`user_inventory`)와는 완전히 분리된 별도 테이블.

현재 게임이 장비 자체도 캐릭터 스프라이트에 시각적으로 반영되지 않는 구조라, 코스튬도 동일하게 "보유 목록"으로만 존재함 — 나중에 스프라이트에 장비/코스튬을 시각적으로 표시하는 기능이 생기면 그때 코스튬이 실제로 "입혀지는" 형태로 확장 가능(자세한 내용은 [`todo.md`](./todo.md)).
