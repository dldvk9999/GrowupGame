# 월드보스

관련 파일: `WorldBossBattle.jsx`, `DungeonSelect.jsx`(월드보스 탭), `worldBoss.js`, migration 033

## 개요

전체 유저가 **공유 체력풀 하나**를 함께 깎는 주간 레이드 콘텐츠. 던전 탭의 "🐉 월드보스" 서브탭.

- **체력은 매주 일요일 00:00(서울시간)에 초기화** — 못 다 깎으면 새 개체로 리셋
- 하루 **3회** 도전 가능(다른 던전과 동일하게 오전 8시 초기화, 실시간 카운트다운은 `useCountdownToDaily8AM` 재사용, [`stages-and-dungeons.md`](./stages-and-dungeons.md))
- 보스 체력은 **매주 일요일 자정** 초기화, "N일 HH:MM:SS 후" 카운트다운(`useCountdownToWeeklyReset`) — 서버 `sync_world_boss`의 일요일 기준 경계 계산과 동일하게 클라이언트가 재현
- **한 판당 제한시간 1분** — 못 잡아도 그동안 입힌 피해는 그대로 공유 체력에 반영(퇴각/시간초과 모두 같은 보고 경로)
- 체력/공격력이 매우 높아서(아래) **4차 전직 정도는 해야 유효타**
- 클리어(체력 0)하면 **그 주에 조금이라도 피해 입힌 유저 전원**에게 7일짜리 "용의 버프" + 헤더 닉네임 그라데이션 + **피해량 비례 골드 보상(우편)**
- 못 잡고 주가 끝나면 각자 누적 피해량 비례 골드를 우편으로 보상

## 보스 스탯

`world_boss_state`에 저장, 매주 새로 생성 시 고정값:

| 항목 | 값 |
|---|---|
| 최대 체력 | 30,000,000 |
| 공격력 | 7,000(035에서 4,500→7,000, 진행 중인 주에도 즉시 반영) |
| 방어력 | 1,200 |

⚠️ 정확한 밸런스는 실제 플레이하며 조정 필요(유저 수/평균 전투력에 따라 "일주일 안에 다 같이 깎을 수 있는지"가 크게 달라짐). `sync_world_boss()`의 `insert` 값만 바꾸면 됨.

## 전투 화면 - "압도적인" 비주얼

일반 전투화면은 적/플레이어가 비슷한 크기로 대칭 배치되지만, 월드보스는 **전용 아레나 레이아웃**(`.worldboss-arena`):

- 보스 아이콘(🐉)이 **화면 전체를 채우는 크기**로 중앙 배치 — `clamp(290px, 34vw, 340px)`(데스크톱, 기존 72px 대비 4배 이상), 모바일은 `clamp(160px, 62vw, 260px)`
- 은은한 확대/축소 애니메이션(`worldboss-loom`) + 보라/주황 이중 그림자로 위압감 연출
- 플레이어는 기존과 동일하게 작게 한쪽 구석 — "거대한 것 vs 나" 크기 대비 의도적 강조

## 전투 방식

`WorldBossBattle.jsx`는 `DungeonBattle.jsx`와 거의 동일한 전투 엔진([`combat.md`](./combat.md)/[`skills.md`](./skills.md))을 쓰지만:

- 몬스터 도감 성장(exp)은 없음 — 순수히 "이번 전투에서 보스에게 입힌 피해"만 추적(`damageDealtRef`)
- 전투는 플레이어 사망(퇴각), 보스 사망(드묾), **1분 제한시간 종료** 중 하나로 끝남 — 셋 다 결과값만 다르고(`'win'`/`'lose'`/`'timeout'`) 서버 보고 경로는 동일
- 종료 시 `report_world_boss_damage(weekKey, damageDealt)`로 데미지를 보고하고 서버 공식 결과(남은 체력, 클리어 여부)를 받아 반영
- **스킬 영역 반응형 압축 (사용자 피드백/요청)** — 월드보스는 스킬을 최대 10개까지 쓸 수 있어(전직 5차) 다른 화면의 90px+ 스킬 버튼이 화면 아래를 너무 많이 차지하는 문제가 있었음. 처음엔 월드보스 전용 클래스(`worldboss-skills-row`)로 항상 46×46px 아이콘 전용 압축을 적용했는데, PC에서도 무조건 압축돼있어 데스크톱에서는 오히려 다른 전투화면(스킬 이름 노출+넓은 배치)보다 정보가 부족해 보인다는 사용자 피드백으로 규칙을 통합함(자세한 배경은 [`ui-and-ux.md`](./ui-and-ux.md)) — 지금은 월드보스도 다른 전투화면과 완전히 동일한 `.skills-row` 클래스를 쓰고, **모바일(≤640px)에서만** 46×46px 압축(아이콘+쿨다운링+단축키, 이름 숨김)이 걸리며 **PC에서는 이름 노출+넓은 배치**로 통일됨. 모바일 실측 기준 스킬 영역 세로 높이 **405px → 98px(76% 절감)**은 그대로 유지

### 서버 검증 (안티치트)

전투 자체는 클라이언트에서 계산되므로([`security.md`](./security.md)), 보고된 데미지를 무조건 안 믿고 **전투력 기준 상한으로 클램프**:

```
1회 전투 최대 인정 데미지 = max(2000, 내_전투력 * 60)
실제 반영 데미지 = min(보고된_데미지, 위_상한, 보스_남은체력)
```

`fetch_my_combat_power()`(PvP와 동일 헬퍼)로 서버가 직접 재계산한 전투력이 기준이라, 클라이언트가 데미지를 조작해도 상한 이상은 반영 안 됨.

⚠️ **세션 검증(보안 패치, 036)**: 처음엔 `report_world_boss_damage`가 실제로 `enter_world_boss`로 입장했는지 전혀 검증 안 해서, devtools로 입장 절차 없이 이 RPC를 무한 반복 호출하면 **하루 3회 제한이 무시되고 보스 체력을 순식간에 소진**시킬 수 있는 치명적 구멍이 있었음(`enter_world_boss`의 `session_id`가 어디에도 저장 안 되는 버려지는 UUID였던 게 원인). `world_boss_sessions` 테이블을 신설해 던전/전직던전과 동일한 세션 검증 적용 — `enter_world_boss`가 실제 세션을 발급/기록, `report_world_boss_damage(p_session_id, p_damage)`는 그 세션이 "존재+본인 것+미사용"일 때만 반영 후 즉시 소모 처리. 세션당 보고는 정확히 1회만 가능해 하루 3회 제한이 실질적으로도 지켜짐.

## 클리어 보상 — 용의 버프

체력이 0 이하가 되면:

1. `world_boss_state.cleared = true`로 마감
2. 그 주 피해량 1 이상 기록된 **모든 유저**의 `dragon_buff_until`을 `now() + 7일`로 설정(즉시 적용, 우편 개념 아닌 유일한 예외)
3. **피해량 비례 골드 보상도 우편으로 발송**(`reward = min(500000, max(300, round(누적피해량/100)))`, `source_key`로 중복 방지) — 미클리어 보상보다 환산율이 후함
4. 이후 7일간 로그인 시마다 `App.jsx`가 `dragon_buff_until > now()`를 확인해:
   - **공격력/방어력을 "지금의" 20배로**(장비/스킬 보유효과까지 반영된 현재 유효 수치 기준, `equipmentBonus`에 19배만큼 추가). 원본 저장 스탯은 안 건드림(버프 종료 시 자동 원복)
   - **헤더 닉네임에 무지개 그라데이션 애니메이션**(`.app-nickname--dragon`)
   - **남은 시간 표시**: 마이페이지 능력치 카드 아래 "N일 M시간 남음"(`formatRemainingTime`). 헤더 🐉 이모지에도 만료시각 `title` 툴팁

## 미클리어 보상

`sync_world_boss()`가 새 주 보스를 만들기 직전, **직전 주가 미클리어면** 그 주 기여자마다:

```
보상 골드 = min(300000, max(100, round(누적_피해량 / 150)))
```

`add_gold` 호출 + 우편함 안내(`source_key`로 중복 방지).

## 이번 주 기여자 TOP 10 (052)

월드보스 패널 하단에 이번 주 상위 10명(`fetchWorldBossTopContributors`).

- `world_boss_contributions`/`profiles` 둘 다 "누구나 조회 가능" RLS라, **PostgREST 관계 임베딩**(`.select('total_damage, profiles(nickname)')`)으로 직접 조회 — security definer 함수 불필요(원래 공개 데이터라 RLS 우회 자체가 불필요)
- 조회 실패해도 `.catch(() => setTopContributors([]))`로 조용히 숨겨지고 나머지 화면은 정상(부분 장애가 전체를 안 깨뜨림)
- ⚠️ **배포 전 자체 검토로 발견/수정한 버그**: 랭킹 화면의 `.leaderboard-row`(6컬럼 그리드)를 그대로 재사용해 3개 자식만 넣었더니 닉네임이 24px 칸에 짜부러지는 레이아웃 붕괴가 있었음 → 전용 flex 클래스(`worldboss-contributor-row`)로 분리해 해결. **다른 화면의 grid row 클래스를 자식 개수가 다른 곳에 재사용하지 말 것** — 컬럼 수가 안 맞으면 에러 없이 레이아웃만 조용히 깨짐

## 역대 참여 여부 조회 (`hasEverParticipatedInWorldBoss`, 053)

`world_boss_contributions`가 공개 RLS라, 특정 유저가 **어떤 주든 한 번이라도 피해를 입힌 적 있는지**를 RPC 없이 직접 조회 가능. "월드보스 첫 참여" 업적(053, [`attendance-and-achievements.md`](./attendance-and-achievements.md)) 판정에 씀 — `fetchMyWorldBossProgress()`의 `myWeekDamage`는 이번 주만 보여줘서 과거 참여자 진행도가 잘못 0으로 보이는 문제 때문에 별도 분리.

## 지연 생성 패턴

우편함/PvP상점과 동일하게 **cron 없이 지연 생성** — `sync_world_boss()`가 화면 진입 시 호출되어 "이번 주 보스가 없으면 생성 + 지난 주 정산"을 처리함. 아무도 접속 안 하는 주가 있으면 다음 접속자가 트리거.

주차 키(`week_key`)는 일요일 날짜 문자열이고, `date_trunc('week', ...)`가 기본 월요일 기준이라 `+1일 후 truncate, -1일`로 일요일 기준 보정.

## 내 순위 표시 (`fetchMyWorldBossRank`)

TOP10 위젯과 별개로 "이번 주 내가 입힌 피해" 옆에 "현재 N위" 표시.

- 공개 RLS로 **"나보다 데미지 높은 행 개수 + 1"을 count 쿼리 하나로** 계산(`fetch_my_rank`와 동일 아이디어, RPC 없이 `.gt('total_damage', myDamage)` 직접 처리)
- 이번 주 아직 참여 안 했으면 순위 표시 생략

## ⚠️ 확인된 사항: 용의 버프는 PvP/랭킹에 미반영(의도로 추정)

용의 버프(20배)는 **클라이언트가 계산하는 실제 몬스터 전투(스테이지/던전/월드보스)에서만** 적용되고, **서버 쪽 PvP(`start_pvp_battle`)와 랭킹(051/057)은 `dragon_buff_until`을 전혀 참조하지 않음**. 버프 받은 유저도 PvP/랭킹에선 평소 전투력 그대로.

**버그로 판단하지 않고 그대로 둔 이유**: PvP/랭킹에도 20배를 반영하면 버프 유저가 매칭 범위(±25%)를 완전히 벗어나거나 무조건 승리해 PvP의 경쟁 의미가 사라짐. 스테이지 전투에선 화끈한 보상 체감을 주되 경쟁 요소는 공정성을 지키려는 의도적 설계로 추정됨. 확인 안 됐지만 반영 쪽으로 임의 변경하면 밸런스를 심각하게 해칠 위험이 커 사용자 승인 없이 손대지 않음.

## 월드보스 누적 피해량 마일스톤 업적 (migration 090, 신규 콘텐츠)
