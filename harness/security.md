# 보안

여러 차례 진행한 보안 감사에서 발견/수정한 취약점과, 완전한 서버 권위 구조는 아직 아니라는 알려진 한계를 정리.

## 009 보안 감사에서 발견/수정한 것

1. **[치명] `add_gold` 직접 호출 가능 → 무제한 골드 획득** — `add_gold(target_user, amount)`가 신원 확인(`auth.uid()=target_user`)과 1회 상한만 검사하고, "이게 실제로 정당한 보상인지"는 전혀 검증하지 않았음. 브라우저 콘솔에서 `supabase.rpc('add_gold', {target_user: 내id, amount: 100000})`을 반복 호출하면 사실상 무한 골드 획득이 가능했음.
   → **수정**: `add_gold`/`spend_gold`의 EXECUTE 권한을 `authenticated`에서 회수해서 client가 직접 호출 자체를 못 하게 잠금. 대신 스테이지클리어/자동사냥/던전 각각의 골드 보상 공식(`calc_stage_gold`, `calc_idle_gold`, `calc_dungeon_gold`)을 SQL로 그대로 옮겨서, 각 전용 RPC가 "무슨 행동을 했는지"에 맞는 금액을 서버가 직접 계산해서 지급하도록 변경. client가 보내는 골드 숫자는 이제 어디서도 신뢰하지 않음.
2. **[중] 자동사냥 골드에 요청 속도 제한 없음** — 자동사냥은 3초마다(현재는 1.5초) 반복 호출되는 구조라, 스크립트로 그 텀보다 빠르게 반복 호출하면 실질적으로 무제한 파밍이 가능했음.
   → **수정**: `grant_idle_reward`에 유저별 최소 2.5초 간격 제한을 서버에 추가.
3. **[중] 던전 보상이 "입장했다는 사실"과 분리되어 있었음** — 던전은 입장만 하루 3회로 막혀있었을 뿐, 승리 보상(`addGold` 직접 호출)은 던전에 실제로 들어갔는지와 무관하게 호출 가능했음.
   → **수정**: `dungeon_sessions` 테이블을 신설해서 "입장 → 그 세션 하나당 보상 1회만 수령" 구조로 변경. `claim_dungeon_reward`는 존재하는 세션, 내 것, 아직 안 받은 것일 때만 지급.
4. **[경] 레이스컨디션 2건** — `use_dungeon_attempt`(하루 3회 체크)와 `update_nickname`(1회 수정 체크)가 "읽고 → 조건 확인 → 쓰기" 순서라서, 동시에 여러 번 요청을 날리면 그 사이 틈으로 제한을 넘길 수 있었음(예: 던전 4회 이상 입장, 닉네임 2번 이상 변경).
   → **수정**: 둘 다 조건을 WHERE절에 포함한 단일 원자적 UPDATE로 재작성해서 동시요청에도 제한이 정확히 지켜지도록 수정.
5. **[경] 명시적 권한 회수 누락** — `mails`, `dungeon_attempts`, `coupons` 등 최근 추가한 테이블들에 RLS는 걸려있었지만, 이전 테이블들처럼 명시적 `revoke insert/update/delete`는 빠져있었음(RLS만으로도 현재는 막혀있었지만, 이중 방어 차원에서 다른 테이블들과 동일하게 명시적으로 회수함).

## 이후 보안 패치

→ **수정**: `claim_mission_reward`에 "미션이 배정된 시각으로부터 최소 20초" 게이트 추가(idle 보상의 2.5초 최소 간격 제한과 동일한 설계). 자세한 내용은 [`guide-missions.md`](./guide-missions.md).

### 027 — 로비채팅 도배 방지

채팅은 RPC가 아니라 client가 테이블에 직접 INSERT하는 구조라, 닉네임 사칭은 트리거로 막혀있었지만(004) 전송 속도 제한이 없어서 봇으로 도배가 가능했음.

→ **수정**: `chat_rate_limit_guard` 트리거로 본인 기준 최소 2초 간격을 강제. 자세한 내용은 [`social-chat.md`](./social-chat.md).

### RPC 작성 시 반복 발생한 버그 패턴 — "column reference X is ambiguous"

`RETURNS TABLE(..., X, ...)`로 정의한 함수는 그 컬럼명(X)이 함수 본문 내에서 암묵적으로 변수처럼도 취급됨. 함수 본문에서 실제 테이블의 동일한 이름 컬럼을 **테이블 별칭 없이 bare로** UPDATE/WHERE/INSERT-ON CONFLICT 등에 쓰면 "이게 리턴 컬럼이야 테이블 컬럼이야"하고 모호해져서 에러가 남.

이 문제가 `draw_skill`(007에서 수정), `draw_equipment`/`draw_equipment_batch`(018/019에서 두 단계에 걸쳐 수정, `ON CONFLICT (user_id, slot)`의 컬럼 목록 표기까지 문제였음), `enter_world_boss`(034에서 수정, `WHERE week_key = ...`), **`claim_attendance`(046에서 신설했다가 배포 후 실사용 중 발견, 056에서 수정)**에서 실제로 발생했음. **046은 이 harness 문서에 패턴이 이미 정리돼있던 이후에 짠 코드인데도 똑같은 실수를 반복했음** — 새 RPC를 작성할 때마다 이 섹션을 실제로 체크리스트처럼 대조하지 않으면 알고 있어도 또 틀릴 수 있다는 방증.

**대응 원칙**: 새 RPC를 짤 때 `RETURNS TABLE`을 쓴다면, 함수 본문 내 모든 SQL 문(특히 UPDATE의 SET/WHERE, INSERT의 ON CONFLICT)에서 테이블 컬럼은 항상 별칭을 붙여서(`ui.slot`처럼) 참조할 것. ON CONFLICT의 컬럼 목록 표기가 불안하면 `ON CONFLICT ON CONSTRAINT 제약이름`처럼 제약 이름으로 지정하는 것도 안전한 대안.

### 036 — 월드보스 무제한 데미지 보고 (가장 심각했던 사례)

`report_world_boss_damage(p_week_key, p_damage)`가 "이 유저가 실제로 `enter_world_boss`로 입장권을 소모하고 전투를 치렀는지"를 전혀 검증하지 않았음. `enter_world_boss`가 반환하는 `session_id`는 어디에도 저장되지 않는 그냥 버려지는 UUID였음.

→ devtools에서 `report_world_boss_damage`를 입장 절차 없이 무한 반복 호출하면, 1회 호출당 데미지는 전투력 기준으로 클램프되지만 **호출 횟수 자체엔 아무 제한이 없어서** 하루 3회 제한이 완전히 무시되고 보스 체력을 순식간에 소진시켜 "일주일 동안 다같이 깎는" 레이드 설계 자체가 무력화될 수 있었음. 지금까지 발견한 것 중 가장 심각한 취약점.

→ **수정(036)**: `world_boss_sessions` 테이블을 신설해서 던전/전직던전과 동일한 세션 검증 패턴 적용. `enter_world_boss`가 실제 세션을 발급/기록하고, `report_world_boss_damage(p_session_id, p_damage)`는 그 세션이 "존재+본인 것+미사용"일 때만 데미지를 반영하고 즉시 소모 처리함. 자세한 내용은 [`world-boss.md`](./world-boss.md).

**교훈**: "입장 제한(하루 N회)"과 "보상 지급(RPC)"이 분리된 2단계 플로우를 설계할 때는, 반드시 **입장 시 발급한 세션을 서버에 영속적으로 저장**하고 보상 지급 RPC가 그 세션의 존재/소유/미사용 여부를 검증해야 함. "클라이언트에 세션 ID를 반환하는 것"과 "그 세션 ID를 서버가 나중에 검증할 수 있는 것"은 완전히 다른 이야기 — 전자만 하고 후자를 빠뜨리면 입장 제한이 사실상 있으나 마나 한 장식이 됨.

### 037 — 전체 재점검에서 발견한 4건 (한번에 패치)

프로젝트 시작부터 끝까지 모든 RPC를 다시 훑어서 찾은 문제들. 심각도 순:

1. **[치명적] `grant_idle_reward`가 chapter/player_level을 클라이언트가 보내는 값 그대로 신뢰**
   자동사냥 골드 지급 RPC가 "지금 몇 챕터인지, 캐릭터 레벨이 몇인지"를 client가 보낸 파라미터로만 계산하고 있었음. devtools에서 `grant_idle_reward(100, 999)`처럼 과장된 값을 2.5초마다(기존 쿨다운 한도) 반복 호출하면, 실제 진행도와 완전히 무관하게 대량의 골드를 무제한 파밍할 수 있었음. `add_gold` 직접호출을 막았던 009 패치의 취지가 이 경로로는 그대로 뚫려있었던 셈.
   → **수정**: chapter는 `stage_progress`에서 실제 클리어한 최고 챕터, level은 `owned_monsters`의 실제 활성 몬스터 레벨로 서버가 직접 계산하도록 변경. 클라이언트가 보내는 파라미터는 이제 완전히 무시됨(함수 시그니처는 유지해서 클라이언트 코드 수정 불필요).

2. **[중요] `claim_dungeon_reward`/`claim_job_dungeon`이 "실제로 이겼는지" 검증을 안 함**
   두 함수 다 "세션이 존재하고, 아직 안 썼는지"만 확인하고 보상을 지급했음 — 전투를 실제로 치렀는지, 이겼는지는 전혀 안 봄. `use_dungeon_attempt`/`start_job_dungeon`으로 세션만 발급받고 바로 `claim`을 호출해도 그대로 성공했음. 특히 **전직 던전은 하루 입장 횟수 제한 자체가 없어서**, 레벨 조건만 채우면 전투 한 번 없이 1~5차 전직을 전부 즉시 완료할 수 있었음(다른 곳보다 훨씬 심각한 버전).
   → **수정**: 세션 생성 시각과 클레임 시각 사이에 최소 시간(일반 던전 2초, 전직 던전 3초)이 지나야만 클레임 가능하도록 게이트 추가. **다만 이건 부분적인 완화책**임 — devtools로 sleep을 넣어 우회하는 것 자체는 여전히 가능함. 진짜 근본 해결은 전투 판정 자체를 서버가 재현/검증하는 구조가 필요한데, 이건 이 프로젝트의 "전투는 클라이언트 계산" 전제([`security.md`](#알려진-한계-완벽한-서버-권위-구조는-아님) 알려진 한계) 자체를 바꿔야 하는 큰 작업이라 이번 패치 범위 밖으로 남겨둠.

3. **[중간] `claim_mail` 레이스컨디션 + 중복 아이템 우편이 클레임을 통째로 실패시키던 버그**
   - 우편 조회 시 `for update` 락이 없어서, 같은 우편을 아주 짧은 간격으로 두 번 요청하면 골드가 두 번 지급될 여지가 있었음 → `for update` 추가로 수정
   - `user_inventory`는 `unique(user_id, item_key)` 제약이 있는데(014), 이미 보유한 아이템이 든 우편(예: `WELCOME2026` 쿠폰의 레어 무기를 이미 뽑기로 갖고 있는 유저)을 받으면 INSERT가 유니크 제약 위반으로 실패해서 **골드까지 포함한 우편 전체가 클레임 실패**했음 → `ON CONFLICT DO UPDATE`로 변경해서, 이미 보유한 아이템이면 뽑기 중복과 동일하게 강화수치 +1로 병합되도록 수정(클레임 자체가 실패하지 않음)

4. **[경미] `redeem_coupon`의 `max_uses` 체크가 읽고-쓰기 분리라 동시요청 시 소량 초과 가능**
   여러 유저가 거의 동시에 마지막 남은 사용횟수의 쿠폰을 입력하면, 각자 "아직 남았다"고 읽은 뒤 둘 다 통과해서 `max_uses`를 살짝 초과할 수 있었음 → 원자적 `UPDATE ... WHERE used_count < max_uses`로 재작성

이번 재점검에서는 001~036 전체 마이그레이션의 모든 `security definer` 함수를 하나씩 다시 훑었고, 이전에 고쳤던 이슈(ambiguous column, 월드보스 세션검증, 미션 쿨다운, PvP 쿨다운, 채팅 도배)들은 재확인 결과 전부 정상 작동 중임을 확인함.

### 038~043 재점검 (골드미션 동적화 작업 중 자체 발견)

- **[경미] 043 작성 시 `claim_mission_reward`를 029 원본에서 복사하면서 스킬 슬롯 상한 계산(`v_slot_limit`)이 041 이전 5단계(Lv.75/50/25/10) 기준으로 되돌아가있던 실수** — 실제 배포 전 자체 재검토에서 발견. 041에서 슬롯 최대치를 5→10으로 늘리며 레벨 구간을 10단계(Lv.220/190/160/130/100/75/50/25/10)로 확장했는데, 바로 다음 마이그레이션인 043이 `claim_mission_reward`를 다시 작성하면서 이 확장을 반영 안 한 채 커밋했으면 `equip_skill_slot` 온보딩 미션 판정과 다음 미션 결정 로직이 옛날 기준으로 되돌아가는 회귀 버그가 될 뻔했음. 배포 전 041과 동일한 10단계 CASE문으로 맞춰서 수정함.
- **교훈**: 어떤 RPC를 "원본을 복사해서 일부만 바꾸는" 방식으로 재정의할 때는, 그 RPC 안에 다른 마이그레이션에서 이미 바뀐 상수/로직(이번 경우 041의 슬롯 상한)이 섞여 있는지 반드시 diff로 확인할 것. 특히 `claim_mission_reward`처럼 여러 마이그레이션(020/021/022/029/043)에 걸쳐 계속 전체 재정의되어 온 함수는 최신 버전을 기준으로 삼되, 그 사이에 다른 함수(`set_skill_loadout` 등)에서만 바뀌고 이 함수엔 아직 반영 안 된 상수가 없는지 교차 확인 필요.
- 042(코스튬 착용)의 `set_costume_loadout`은 보유 검증(`count(*) = array_length`)과 슬롯 중복 검증(`count(distinct slot) = array_length`) 로직을 재점검함 — 배열에 중복 원소를 넣어 보유 검증을 우회하려는 시도는 `count(*)`가 distinct 매칭 row 수만 세기 때문에 배열 길이와 불일치가 나서 막힘. 정상 작동 확인.
- 038(PvP 쿨다운)/039(스킬레벨 상한)/040(일반몹 상향)/041(스킬슬롯 상한)은 기존 검증된 함수 본문을 그대로 복사해서 상수만 바꾼 구조라 별도 취약점 없음.

### 044 — equipment_gacha_progress 컬럼명 오타로 미션 클레임 전체 장애

배포 전 자체 점검에서는 슬롯 상한 회귀만 잡았으나, **실제 배포 후 `equipment_gacha_progress.total_draws`를 `draws`로 잘못 참조한 오타를 놓쳤음** — 테이블 정의(017)를 다시 대조하지 않고 이름을 추정해서 쓴 게 원인. 그 결과 `spend_gold` 순번이 아니어도(어떤 미션이든) 완료 버튼을 누르면 `claim_mission_reward` 내부에서 `column "draws" does not exist` 에러가 나서 **미션 클레임 자체가 전면 불가능**해지는 장애로 이어짐.

→ **수정(044)**: `claim_mission_reward`를 다시 재정의해서 `total_draws`로 정정.

**교훈**: 다른 테이블의 컬럼을 참조하는 새 로직을 짤 때는 해당 테이블을 처음 만든 마이그레이션(`create table`)을 직접 열어서 컬럼명을 대조할 것 — 이름을 추정하거나 비슷한 다른 컬럼명(`profiles.total_skill_draws`)에서 유추하면 이런 오타가 재발함. 가능하면 로컬에서 실제 쿼리를 실행해보고 배포하는 습관이 필요함.

### 046~051 정기 보안점검 (게임빌드업 세션, push 3회 이후 정례 점검)

이번 세션에서 새로 추가한 6개 마이그레이션(출석체크/업적/랭킹/무료뽑기/칭호/PvP장비보너스)을 전수 재점검함:

- **RLS/권한**: 새로 생긴 테이블 3개(`attendance_state`/`achievement_claims`/`daily_free_draw_state`) 전부 RLS 활성화 + 본인만 SELECT 정책 + insert/update/delete는 authenticated로부터 revoke — 기존 패턴과 동일하게 RPC(security definer)로만 변경 가능함을 확인
- **`auth.uid()` 검증**: 모든 상태변경 RPC가 `auth.uid() is null` 체크로 시작하고, 대상 유저를 파라미터로 안 받고 항상 `auth.uid()` 자기 자신만 조작함(권한 상승/타인 계정 조작 경로 없음)
- **SQL 인젝션**: 동적 SQL(`EXECUTE format(...)`) 사용 없음, 전부 파라미터화된 비교/CASE문
- **`claim_daily_free_draw`의 "임시버퍼→원상복구" 방식 재검증**: 시작 시점 잔액을 `for update`로 잠근 채 캡처 → 큰 버퍼 지급 → 실제 뽑기(정상 비용 차감) → 캡처해둔 잔액으로 강제 리셋. 중간에 뽑기 RPC가 예외를 던지면 트랜잭션 전체가 롤백되어 무료뽑기가 소모되지 않고 되돌아감(부분 상태 남지 않음) — 안전. `for update` 락 덕분에 동시 중복호출도 직렬화되어 이중 무료뽑기 불가능
- **`fetch_leaderboard`/`fetch_my_rank`**: 전체 유저를 조회하는 security definer 함수지만 닉네임/레벨/전직단계/속성/전투력/칭호만 반환하고 골드·이메일 등 민감정보 노출 없음
- **`calc_equipped_stat_bonus`**: security definer가 아닌 일반 함수라 클라이언트가 임의로 `p_user_id`(타인)를 넣어 직접 호출해도, `user_inventory`의 RLS(본인만 조회)가 SECURITY INVOKER 컨텍스트에 그대로 적용되어 타인 데이터는 0으로 나옴 — 정보 노출 없음(security definer 함수들 내부에서 호출될 때만 그 함수의 상승된 권한으로 실제 데이터를 읽음)
- **함수 반환타입 배포 오류**: 위에서 이미 다룬 `fetch_leaderboard` DROP FUNCTION 이슈 외에, 이번 세션에서 재정의한 `draw_skill`/`draw_equipment`/`claim_mission_reward`/`start_pvp_battle`/`set_skill_loadout` 등은 전부 이전 버전과 반환 컬럼 구성이 동일한지 diff로 전수 확인 완료(문제 없음)

**결론**: 이번 점검에서 새로 발견된 취약점은 없음(위 배포 오류는 보안이 아닌 배포 안정성 이슈였고 이미 패치 완료).

### 052~054 재점검 (게임빌드업 세션 계속, 두 번째 정기 점검)

- **052(`create_starter_monster` 재정의)**: 원본(006)부터 `auth.uid() is null` 명시적 체크가 없었음(선행 발견, 내가 만든 문제 아님) — 다만 `owned_monsters.user_id`가 NOT NULL이라 미인증 상태로는 첫 INSERT부터 제약조건 위반으로 실패하므로 실질적 보안 구멍은 아니고, 에러 메시지가 덜 친절할 뿐. 새로 추가한 환영 우편 INSERT도 그 이전 단계가 실패하면 도달 자체가 안 되므로 안전
- **053(업적 2종 추가)**: `world_boss_participate` 체크가 `where user_id = auth.uid()`로 정확히 스코프됨, 새로 추가된 CASE 분기 2개 다 타인 데이터 접근 경로 없음
- **054(쿠폰 시딩)**: 함수가 아니라 정적 데이터 INSERT라 새로운 공격 표면 없음. 기존에 이미 검증된 `redeem_coupon`(037에서 원자적 UPDATE로 레이스컨디션 수정 완료)을 그대로 타므로 안전
- **`hasEverParticipatedInWorldBoss(userId)` 클라이언트 직접조회**: `world_boss_contributions`가 애초부터(033) "누구나 조회 가능" RLS라, 타인의 `userId`를 넣어 호출해도 어차피 이미 공개된 데이터(랭킹 TOP10 위젯도 동일 테이블을 전체 공개 조회함)라 새로운 정보 노출 아님
- **로그아웃 시 게임데이터 state 전체 초기화 리팩토링**: `setSession`/`setStage` 순서나 부수효과 없음 확인, 단순 초기값 재설정이라 무한루프/부작용 없음

**결론**: 새로 발견된 취약점 없음.

### 055~이후 추가분 3차 정기 점검 (PvP전적/럭키보너스/Presence/강화바)

## 알려진 한계 (완벽한 서버 권위 구조는 아님)

⚠️ **037 재점검에서 재확인된 핵심 한계**: `claim_dungeon_reward`/`claim_job_dungeon`은 여전히 "전투에서 실제로 이겼는지"를 완전히 검증하지 못함(최소 시간 게이트만 있음). 근본적으로는 전투 판정을 서버가 직접 재현/검증해야 완전히 막을 수 있는데, 이건 아래 항목들과 같은 성격의(그리고 이 프로젝트에서 가장 큰) 구조적 한계임.

- 전투 데미지 계산/스킬 배율/장비 보너스 적용은 **클라이언트(브라우저)에서 계산**됨. 서버는 "최종 결과값(레벨/스탯/골드)"이 물리적으로 가능한 범위 안에 있는지만 사후 검증함(상한선 방식). 즉 "있을 수 없는 큰 값"은 막지만, 그럴듯한 범위 내의 정교한 조작까지 100% 차단하진 못함.
- 완전한 서버 권위 구조로 가려면 전투 판정 자체를 서버(Edge Function 등)에서 수행해야 함 — 현재는 프로토타입 단계로 판단해 이 정도 수준에서 타협함.
- **정기 우편은 실제 cron이 아니라 "지연 생성" 방식**임 — 08/12/19시가 지난 뒤 유저가 우편함에 들어와야(`sync_daily_mails` 호출) 그 시점에 우편이 생성됨. pg_cron으로 실제 예약 발송을 구현할 수도 있으나(Supabase 플랜에 따라 지원 여부 다름) 지금은 더 안정적인 지연생성 방식으로 구현함. PvP 상점(`sync_pvp_shop`)도 동일한 패턴.
- **PvP 전투력 계산에 장비/스킬 보유효과가 포함되지 않음** — 서버에서 매번 인벤토리를 조인해서 계산하는 부담을 피하려는 의도적 단순화. 장비/스킬 비중이 커지면 나중에 반영 검토 필요 (자세한 내용은 [`pvp.md`](./pvp.md))
- **월드보스도 동일한 "클라이언트 계산 + 서버 상한 클램프" 패턴**을 씀 — 전투 자체는 클라이언트에서 진행되고, 종료 후 보고된 데미지를 `전투력×60`(최소 2000) 상한으로 클램프해서 공유 체력에 반영함(`report_world_boss_damage`). 완벽한 서버권위 시뮬레이션은 아니지만, 전투력 기준 상한이 있어서 터무니없는 값 보고는 막힘 (자세한 내용은 [`world-boss.md`](./world-boss.md))
