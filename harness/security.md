# 보안

여러 차례 진행한 보안 감사에서 발견/수정한 취약점과, 완전한 서버 권위 구조는 아직 아니라는 알려진 한계를 정리.

## 009 보안 감사에서 발견/수정한 것

1. **[치명] `add_gold` 직접 호출 가능 → 무제한 골드 획득** — `add_gold(target_user, amount)`가 신원 확인과 1회 상한만 검사하고 "정당한 보상인지"는 전혀 검증 안 함. 브라우저 콘솔에서 반복 호출하면 사실상 무한 골드 획득 가능했음.
   → **수정**: `add_gold`/`spend_gold`의 EXECUTE 권한을 `authenticated`에서 회수해 client 직접 호출 자체를 차단. 대신 스테이지클리어/자동사냥/던전 각각의 골드 공식(`calc_stage_gold` 등)을 SQL로 옮겨서, 각 전용 RPC가 서버에서 직접 계산해 지급. client가 보내는 골드 숫자는 이제 어디서도 신뢰 안 함
2. **[중] 자동사냥 골드에 요청 속도 제한 없음** — 반복 호출로 무제한 파밍 가능했음
   → **수정**: `grant_idle_reward`에 유저별 최소 2.5초 간격 제한 추가
3. **[중] 던전 보상이 "입장했다는 사실"과 분리되어 있었음** — 승리 보상이 실제 입장 여부와 무관하게 호출 가능했음
   → **수정**: `dungeon_sessions` 신설로 "입장 → 세션당 보상 1회만" 구조로 변경
4. **[경] 레이스컨디션 2건** — `use_dungeon_attempt`/`update_nickname`이 "읽고→확인→쓰기" 순서라 동시요청으로 제한을 넘길 수 있었음
   → **수정**: 조건을 WHERE절에 포함한 단일 원자적 UPDATE로 재작성
5. **[경] 명시적 권한 회수 누락** — 최근 추가 테이블들에 명시적 `revoke insert/update/delete`가 빠져있었음(RLS로 이미 막혀있었지만 이중 방어 차원에서 명시적 회수 추가)

### RPC 작성 시 반복 발생한 버그 패턴 — "column reference X is ambiguous"

`RETURNS TABLE(..., X, ...)`로 정의한 함수는 그 컬럼명(X)이 본문 내에서 암묵적으로 변수처럼도 취급됨. 실제 테이블의 동일 이름 컬럼을 **별칭 없이 bare로** UPDATE/WHERE/INSERT-ON CONFLICT 등에 쓰면 모호해져 에러가 남.

이 패턴이 `draw_skill`(007), `draw_equipment`/`_batch`(018/019, ON CONFLICT 컬럼 목록 표기까지 문제), `enter_world_boss`(034), **`claim_attendance`(046에서 신설, 배포 후 실사용 중 발견해 056에서 수정)**에서 실제 발생. **046은 이 패턴이 이미 문서화된 이후에 짠 코드인데도 똑같은 실수를 반복함** — 체크리스트처럼 대조하지 않으면 알고 있어도 또 틀릴 수 있다는 방증.

**대응 원칙**: `RETURNS TABLE`을 쓰는 함수는 본문 내 모든 SQL 문에서 테이블 컬럼에 항상 별칭을 붙일 것. ON CONFLICT 표기가 불안하면 `ON CONFLICT ON CONSTRAINT 제약이름`도 안전한 대안.

### 036 — 월드보스 무제한 데미지 보고 (가장 심각했던 사례)

`report_world_boss_damage(p_week_key, p_damage)`가 실제로 `enter_world_boss`로 입장했는지 전혀 검증 안 함. 반환된 `session_id`가 어디에도 저장 안 되는 버려지는 UUID였음.

→ devtools로 입장 절차 없이 무한 반복 호출하면, 1회 데미지는 전투력 기준으로 클램프되지만 **호출 횟수 자체엔 제한이 없어** 하루 3회 제한이 무시되고 보스 체력을 순식간에 소진시켜 레이드 설계 자체가 무력화될 수 있었음. 지금까지 발견한 것 중 가장 심각한 취약점.

→ **수정(036)**: `world_boss_sessions` 신설로 던전/전직던전과 동일한 세션 검증 패턴 적용. `enter_world_boss`가 실제 세션을 발급/기록, `report_world_boss_damage(p_session_id, p_damage)`는 그 세션이 "존재+본인것+미사용"일 때만 반영 후 즉시 소모 처리. [`world-boss.md`](./world-boss.md).

**교훈**: "입장 제한"과 "보상 지급"이 분리된 2단계 플로우는 반드시 **입장 시 발급한 세션을 서버에 영속 저장**하고 보상 지급 RPC가 그 세션의 존재/소유/미사용 여부를 검증해야 함. 세션 ID를 클라이언트에 반환만 하고 서버가 나중에 검증 못 하면 입장 제한이 있으나 마나 한 장식이 됨.

### 037 — 전체 재점검에서 발견한 4건 (한번에 패치)

프로젝트 전체 RPC를 다시 훑어서 찾은 문제들. 심각도 순:

1. **[치명적] `grant_idle_reward`가 chapter/player_level을 클라이언트가 보내는 값 그대로 신뢰**
   devtools에서 과장된 값을 2.5초마다 반복 호출하면 실제 진행도와 무관하게 골드를 무제한 파밍할 수 있었음. `add_gold` 직접호출을 막았던 009 패치의 취지가 이 경로로 뚫려있었던 셈.
   → **수정**: chapter는 `stage_progress`의 최고 클리어 챕터, level은 실제 활성 몬스터 레벨로 서버가 직접 계산. 클라이언트 파라미터는 완전히 무시(시그니처는 유지해 클라이언트 수정 불필요)

2. **[중요] `claim_dungeon_reward`/`claim_job_dungeon`이 "실제로 이겼는지" 검증을 안 함**
   "세션 존재+미사용"만 확인하고 지급했음 — 세션만 발급받고 바로 클레임해도 성공. 특히 **전직 던전은 하루 입장 제한 자체가 없어** 레벨 조건만 채우면 전투 없이 1~5차 전직을 즉시 완료할 수 있었음(더 심각한 버전).
   → **수정**: 세션 생성~클레임 사이 최소 시간(일반 던전 2초, 전직 던전 3초) 게이트 추가. **부분적 완화책**임 — sleep으로 우회하는 것 자체는 여전히 가능. 근본 해결은 전투 판정을 서버가 재현/검증하는 구조가 필요한데, "전투는 클라이언트 계산"이라는 전제 자체를 바꿔야 하는 큰 작업이라 범위 밖으로 남김

3. **[중간] `claim_mail` 레이스컨디션 + 중복 아이템 우편이 클레임을 통째로 실패시키던 버그**
   - `for update` 락 없어 짧은 간격 중복 요청 시 골드 이중지급 여지 → `for update` 추가
   - `user_inventory`의 `unique(user_id, item_key)` 제약(014) 때문에 이미 보유한 아이템이 든 우편을 받으면 INSERT 실패로 **골드까지 클레임 전체가 실패** → `ON CONFLICT DO UPDATE`로 뽑기 중복과 동일하게 강화수치 +1 병합

4. **[경미] `redeem_coupon`의 `max_uses` 체크가 읽고-쓰기 분리라 동시요청 시 소량 초과 가능**
   여러 유저가 마지막 사용횟수를 거의 동시에 입력하면 초과 가능 → 원자적 `UPDATE ... WHERE used_count < max_uses`로 재작성

이 재점검에서 001~036 전체의 모든 `security definer` 함수를 다시 훑었고, 이전에 고쳤던 이슈들은 재확인 결과 전부 정상 작동 중임을 확인함.

### 038~043 재점검 (골드미션 동적화 작업 중 자체 발견)

- **[경미] 043 작성 시 `claim_mission_reward`를 029 원본에서 복사하면서 스킬 슬롯 상한 계산(`v_slot_limit`)이 041 이전 5단계(Lv.75/50/25/10) 기준으로 되돌아가있던 실수** — 실제 배포 전 자체 재검토에서 발견. 041에서 슬롯 최대치를 5→10으로 늘리며 레벨 구간을 10단계(Lv.220/190/160/130/100/75/50/25/10)로 확장했는데, 바로 다음 마이그레이션인 043이 `claim_mission_reward`를 다시 작성하면서 이 확장을 반영 안 한 채 커밋했으면 `equip_skill_slot` 온보딩 미션 판정과 다음 미션 결정 로직이 옛날 기준으로 되돌아가는 회귀 버그가 될 뻔했음. 배포 전 041과 동일한 10단계 CASE문으로 맞춰서 수정함.
- **교훈**: 어떤 RPC를 "원본을 복사해서 일부만 바꾸는" 방식으로 재정의할 때는, 그 RPC 안에 다른 마이그레이션에서 이미 바뀐 상수/로직(이번 경우 041의 슬롯 상한)이 섞여 있는지 반드시 diff로 확인할 것. 특히 `claim_mission_reward`처럼 여러 마이그레이션(020/021/022/029/043)에 걸쳐 계속 전체 재정의되어 온 함수는 최신 버전을 기준으로 삼되, 그 사이에 다른 함수(`set_skill_loadout` 등)에서만 바뀌고 이 함수엔 아직 반영 안 된 상수가 없는지 교차 확인 필요.
- 042(코스튬 착용)의 `set_costume_loadout`은 보유 검증(`count(*) = array_length`)과 슬롯 중복 검증(`count(distinct slot) = array_length`) 로직을 재점검함 — 배열에 중복 원소를 넣어 보유 검증을 우회하려는 시도는 `count(*)`가 distinct 매칭 row 수만 세기 때문에 배열 길이와 불일치가 나서 막힘. 정상 작동 확인.
- 038(PvP 쿨다운)/039(스킬레벨 상한)/040(일반몹 상향)/041(스킬슬롯 상한)은 기존 검증된 함수 본문을 그대로 복사해서 상수만 바꾼 구조라 별도 취약점 없음.

### 044 — equipment_gacha_progress 컬럼명 오타로 미션 클레임 전체 장애

배포 전 자체 점검에서는 슬롯 상한 회귀만 잡았으나, **실제 배포 후 `equipment_gacha_progress.total_draws`를 `draws`로 잘못 참조한 오타를 놓쳤음** — 테이블 정의(017)를 다시 대조하지 않고 이름을 추정해서 쓴 게 원인. 그 결과 `spend_gold` 순번이 아니어도(어떤 미션이든) 완료 버튼을 누르면 `claim_mission_reward` 내부에서 `column "draws" does not exist` 에러가 나서 **미션 클레임 자체가 전면 불가능**해지는 장애로 이어짐.

→ **수정(044)**: `claim_mission_reward`를 다시 재정의해서 `total_draws`로 정정.

**교훈**: 다른 테이블의 컬럼을 참조하는 새 로직을 짤 때는 해당 테이블을 처음 만든 마이그레이션(`create table`)을 직접 열어서 컬럼명을 대조할 것 — 이름을 추정하거나 비슷한 다른 컬럼명(`profiles.total_skill_draws`)에서 유추하면 이런 오타가 재발함. 가능하면 로컬에서 실제 쿼리를 실행해보고 배포하는 습관이 필요함.

### 046~051 정기 보안점검 (게임빌드업 세션, push 3회 이후 정례 점검)

이번 세션에서 새로 추가한 6개 마이그레이션(출석체크/업적/랭킹/무료뽑기/칭호/PvP장비보너스)을 전수 재점검함:

- **RLS/권한**: 새로 생긴 테이블 3개(`attendance_state`/`achievement_claims`/`daily_free_draw_state`) 전부 RLS 활성화 + 본인만 SELECT 정책 + insert/update/delete는 authenticated로부터 revoke — 기존 패턴과 동일하게 RPC(security definer)로만 변경 가능함을 확인
- **`auth.uid()` 검증**: 모든 상태변경 RPC가 `auth.uid() is null` 체크로 시작하고, 대상 유저를 파라미터로 안 받고 항상 `auth.uid()` 자기 자신만 조작함(권한 상승/타인 계정 조작 경로 없음)
- **SQL 인젝션**: 동적 SQL(`EXECUTE format(...)`) 사용 없음, 전부 파라미터화된 비교/CASE문
- **`claim_daily_free_draw`의 "임시버퍼→원상복구" 방식 재검증**: 시작 시점 잔액을 `for update`로 잠근 채 캡처 → 큰 버퍼 지급 → 실제 뽑기(정상 비용 차감) → 캡처해둔 잔액으로 강제 리셋. 중간에 뽑기 RPC가 예외를 던지면 트랜잭션 전체가 롤백되어 무료뽑기가 소모되지 않고 되돌아감(부분 상태 남지 않음) — 안전. `for update` 락 덕분에 동시 중복호출도 직렬화되어 이중 무료뽑기 불가능
- **`fetch_leaderboard`/`fetch_my_rank`**: 전체 유저를 조회하는 security definer 함수지만 닉네임/레벨/전직단계/속성/전투력/칭호만 반환하고 골드·이메일 등 민감정보 노출 없음
- **`calc_equipped_stat_bonus`**: security definer가 아닌 일반 함수라 클라이언트가 임의로 `p_user_id`(타인)를 넣어 직접 호출해도, `user_inventory`의 RLS(본인만 조회)가 SECURITY INVOKER 컨텍스트에 그대로 적용되어 타인 데이터는 0으로 나옴 — 정보 노출 없음(security definer 함수들 내부에서 호출될 때만 그 함수의 상승된 권한으로 실제 데이터를 읽음)
- **함수 반환타입 배포 오류**: 위에서 이미 다룬 `fetch_leaderboard` DROP FUNCTION 이슈 외에, 이번 세션에서 재정의한 `draw_skill`/`draw_equipment`/`claim_mission_reward`/`start_pvp_battle`/`set_skill_loadout` 등은 전부 이전 버전과 반환 컬럼 구성이 동일한지 diff로 전수 확인 완료(문제 없음)

**결론**: 이번 점검에서 새로 발견된 취약점은 없음(위 배포 오류는 보안이 아닌 배포 안정성 이슈였고 이미 패치 완료).

### 052~054 재점검 (게임빌드업 세션 계속, 두 번째 정기 점검)

### 104 — 던전 세션 stage 체크 제약 500층 미반영 (사용자 제보 버그)

**증상**: "경험치 던전 도전하기" 클릭 시 `new row for relation "dungeon_sessions" violates check constraint "dungeon_sessions_stage_check"` 에러.

**원인**: 009에서 `dungeon_sessions.stage`에 `check (stage between 1 and 10)`을 걸었는데, 079가 던전 최고 층수를 10→500으로 올리며 `use_dungeon_attempt()` **함수 로직만** 고치고 이 **테이블 제약조건**은 안 건드림. 진행 층수(`cleared_stage+1`)가 11층 이상인 유저는 세션 INSERT 자체가 거부되어 던전 입장이 아예 막혀 있었음(079 이후 줄곧 존재했을 가능성이 높은 오래된 회귀 버그로 추정).

**교훈**: `job_dungeon_sessions.tier` 체크 제약은 4차/5차 전직 추가(021/029) 때마다 `drop constraint`+`add constraint`로 매번 같이 갱신됐는데(같은 세션 테이블 패턴), `dungeon_sessions.stage`는 079에서 이 단계가 누락됨. **최고 스테이지/층수/티어 상한을 올리는 마이그레이션은 함수 내부 로직뿐 아니라 관련 테이블의 `check` 제약도 항상 같이 검색해서 확인할 것** — harness에 이미 있는 "DROP FUNCTION 누락 탐지"와 같은 성격의 "체크 제약 미반영" 패턴으로 기록.

**수정(104)**: `dungeon_sessions_stage_check`를 `stage between 1 and 500`으로 재정의. `drop constraint if exists` + `add constraint`라 기존 데이터에 영향 없음(기존 행은 전부 1~10 범위라 재검증 통과).

### 106 — [긴급/치명적] 오프라인 골드 보상 무한 파밍 취약점 (자체 재검토 중 발견, 배포 후 발견)

**배경**: 103(오프라인 골드 보상)을 만들 때, 102(복귀 보상)가 쓰는 `profiles.last_login_at`을 그대로 재사용하되 "오프라인 보상 함수는 읽기만 하고 갱신은 comeback 함수가 전담"하도록 설계했음(두 함수의 실행 순서를 클라이언트가 보장하는 구조).

**문제**: 이 설계에선 `claim_offline_gold_reward()` 자체는 `last_login_at`을 **전혀 갱신하지 않음**. 즉 정상적인 로그인 플로우(App.jsx)를 거치지 않고 브라우저 devtools에서 `supabase.rpc('claim_offline_gold_reward')`를 반복 호출하면, 기준 시각(`last_login_at`)이 절대 움직이지 않아 **같은 오프라인 보상을 무한히 반복 지급**받을 수 있었음. 5분 최소 게이트가 있었지만 이건 "5분 미만이면 0원"일 뿐 반복 호출 자체를 막지는 못했고, 갱신 주체가 없으니 6번째 호출이든 6000번째 호출이든 매번 같은(또는 더 큰) 금액이 그대로 다시 지급됨. 036(월드보스 무제한 데미지)/044(미션 클레임 장애)와 같은 급의 치명도로 판단해 즉시 수정.

**교훈**: `grant_idle_reward`(009)는 처음부터 `last_idle_reward_at`을 **자기 자신이 매 호출마다 갱신**하는 자체완결형 쿨다운 구조였음(2.5초 게이트). 103을 만들 때 이 확립된 패턴을 그대로 따르지 않고 "다른 함수가 갱신을 대신 해주는" 교차의존 구조로 설계한 게 근본 원인. **보상/재화 지급 RPC는 예외 없이 "자기가 쓰는 쿨다운/체크포인트 컬럼을 자기 자신이 매 호출마다 갱신"하는 자체완결형으로 만들 것** — 다른 함수의 호출을 전제로 한 교차의존 쿨다운은 클라이언트가 그 전제(호출 순서)를 지킨다는 보장이 없어 원천적으로 안전하지 않음.

**수정(106)**: `profiles.last_offline_claim_at`을 신규 전용 컬럼으로 분리. `claim_offline_gold_reward`가 이 컬럼을 `for update`로 잠근 뒤 지급 여부와 무관하게 **항상 자기 자신을 갱신**하도록 수정(add_gold 판정 전에 UPDATE부터 실행). `last_login_at`(102, 복귀 보상 전용)은 그대로 유지 — 두 체크포인트가 이제 완전히 독립적이라 호출 순서 무관하게 안전함.

### 110 — [긴급/치명적] 월드보스 기여자 보상 지급 크래시 + 이중지급 (신규 기능 작업 중 자체 발견)

**발견 경위**: 월드보스 개인 최고기록 기능을 추가하려고 `report_world_boss_damage`(036) 원문을 다시 읽던 중, 클리어/미클리어 보상 루프에서 기여자마다 `perform public.add_gold(v_contrib.user_id, v_reward)`를 직접 호출하는 걸 발견.

**버그 1 [치명적] 크래시**: `add_gold(target_user, amount)`(003에서 신설, 030이 최신)는 `auth.uid() <> target_user`면 즉시 `raise exception 'not authorized'`로 막는 구조(009에서 EXECUTE도 `authenticated`에서 회수해 반드시 다른 security definer 함수 내부에서만 호출 가능). 그런데 월드보스 클리어 루프는 "방금 마지막 타격을 넣은 사람"(`auth.uid()`) 컨텍스트로 실행되면서 **자신이 아닌 다른 기여자들**의 `user_id`로 `add_gold`를 호출함 — 기여자가 2명 이상이면 자기 자신이 아닌 첫 기여자 차례에서 곧바로 예외가 터지고, 예외 핸들러가 없으니 **트랜잭션 전체가 롤백**됨(체력 반영/`cleared` 플래그/용의 버프/보상우편 전부 무효화, 호출자는 에러만 받음). 월드보스는 설계상 여러 유저가 함께 깎는 콘텐츠라 기여자 2명 이상은 사실상 항상 발생하는 상황이므로, **실제로는 아무도 진짜 "클리어"를 성공적으로 끝내보지 못했을 가능성이 높은 치명적 버그**로 판단. `sync_world_boss()`의 미클리어 주간보상 루프도 완전히 동일한 패턴이라, 이전 주가 미클리어 상태로 끝났고 기여자가 2명 이상이면 **새 주 보스 생성(루프 뒤에 있는 INSERT)까지 막힐 수 있는** 구조였음.

**버그 2 [경미] 이중지급**: 설령 기여자가 1명뿐이라 크래시를 피하더라도(그 1명이 곧 `auth.uid()`라 인가 통과), `add_gold`로 즉시 골드를 지급한 뒤 **같은 금액을 `gold_amount`로 담은 우편도 함께 발송**하고 있어서, 그 우편을 나중에 `claim_mail`로 열어보면(037이 실제로 `gold_amount`를 더해줌을 확인) 같은 보상을 정확히 두 번 받게 되는 부차적 문제도 함께 존재.

**교훈**: 다른 보상 경로(던전완주보너스 064/탑마일스톤 075/친구추천 065/코스튬 등)는 전부 "우편으로만" 지급하는 단일 경로인데, 월드보스만 "즉시지급 + 우편"이 섞여 있었음. **보상은 반드시 하나의 경로로만 지급할 것** — 특히 `add_gold`처럼 `auth.uid() = target_user` 인가를 요구하는 헬퍼는 **호출자 본인 몫에만 쓸 것**이고, 여러 명(기여자 목록 등)에게 동시에 지급해야 하는 루프에서는 절대 쓰면 안 됨(우편 INSERT처럼 인가 제약이 없는 경로를 써야 함).

**수정(110)**: `report_world_boss_damage`/`sync_world_boss` 둘 다에서 `add_gold` 직접 호출을 제거하고 우편 발송(수령 시 1회 지급)만 남김. 두 함수 모두 반환 타입 그대로라 DROP FUNCTION 불필요.

**전수감사**: 전체 마이그레이션(001~111)의 `perform public.add_gold(...)` 호출부를 grep으로 전부 나열해 재확인 — 월드보스(033/035/036, 지금은 110으로 수정됨) 외에는 **예외 없이 전부 `auth.uid()`를 대상으로만 호출**하고 있음을 확인. 즉 이 버그 패턴(다른 유저 몫을 `add_gold`로 직접 지급하려다 크래시)은 이번에 고친 것이 유일한 사례였고, 다른 곳에 잠재된 동일 패턴은 없음.
### 60차 점검 — push 4회 시점 선제점검 (107 PvP보상강화, 105 주말보너스, 스킬영역 CSS)

원래는 "push 5회마다" 주기지만, 106에서 치명적 취약점이 직접 나온 직후라 다음 push(5회차) 전에 한 번 더 미리 훑음(선제적 조기점검).

**105(주말골드보너스)**: `grant_idle_reward` 반환타입 그대로(DROP 불필요), `last_idle_reward_at` 자체갱신 2.5초 쿨다운은 그대로 보존 확인(diff로 105 vs 062 비교, 배율 로직만 추가됐음을 재확인). 배율 중첩(주말1.5배×골든3배=4.5배) 최종 500만 클램프로 폭주 방지.

**107(PvP 실유저보상강화)**: `start_pvp_battle` 반환타입 그대로(DROP 불필요). 이 함수는 103/106과 달리 "패시브 시간경과 체크포인트"가 아니라 **호출 1회 = 실제 대결 1판**(승패가 매번 새로 랜덤 판정됨) 구조라, 반복 호출해도 매번 진짜 승부가 나고 `last_pvp_battle_at` 2초 쿨다운도 두 분기(승/패) 모두에서 그대로 자체갱신됨 — 106과 같은 "체크포인트 미갱신" 패턴이 없음을 확인. 패배 시 위로보상 추가가 `pvp_currency`를 깎는 게 아니라 더하기만 하므로 음수 잔액 등 부작용 없음.

**스킬영역 CSS 통일**: 서버/마이그레이션 변경 없음(순수 클라이언트 CSS). `skill-btn` 클래스가 4개 전투화면 외 다른 곳(스킬편성/뽑기 화면)에서 재사용되지 않는지 grep으로 확인 — 없음, 의도치 않은 영향 없음.

**전체 마이그레이션(001~107) 재스캔**: "column reference ambiguous"/"DROP FUNCTION 누락" 자동 스캐너 재실행, 0건. 046(기수정) 외 이슈 없음(변동 없음).

**106 사후 재확인**: 이번 세션에서 새로 만든 모든 RPC(102/103·106/105/107)를 "체크포인트 자체완결형인가"라는 단일 기준으로 재점검(106의 교훈을 즉시 적용) — 102는 원래부터 자체갱신이라 안전, 103은 106에서 수정 완료, 105/107은 처음부터 기존 자체갱신 함수(`grant_idle_reward`/`start_pvp_battle`)를 재정의한 것이라 애초에 이 패턴 위험이 없었음. 신규로 이 패턴을 위반하는 함수 없음을 확인.

## 알려진 한계 (완벽한 서버 권위 구조는 아님)

⚠️ **037 재점검에서 재확인된 핵심 한계**: `claim_dungeon_reward`/`claim_job_dungeon`은 여전히 "전투에서 실제로 이겼는지"를 완전히 검증하지 못함(최소 시간 게이트만 있음). 근본적으로는 전투 판정을 서버가 직접 재현/검증해야 완전히 막을 수 있는데, 이건 아래 항목들과 같은 성격의(이 프로젝트에서 가장 큰) 구조적 한계임.

- 전투 데미지 계산/스킬 배율/장비 보너스 적용은 **클라이언트(브라우저)에서 계산**됨. 서버는 "최종 결과값(레벨/스탯/골드)"이 물리적으로 가능한 범위 안에 있는지만 사후 검증함(상한선 방식) — "있을 수 없는 큰 값"은 막지만 그럴듯한 범위 내 정교한 조작까지 100% 차단은 못 함
- 완전한 서버 권위 구조로 가려면 전투 판정 자체를 서버(Edge Function 등)에서 수행해야 함 — 현재는 프로토타입 단계로 판단해 이 수준에서 타협함
- **정기 우편은 실제 cron이 아니라 "지연 생성" 방식**임 — 08/12/19시가 지난 뒤 유저가 우편함에 들어와야(`sync_daily_mails`) 그 시점에 생성됨. pg_cron으로 실제 예약 발송도 가능하나(Supabase 플랜별 지원 여부 다름) 더 안정적인 지연생성으로 구현함. PvP 상점(`sync_pvp_shop`)도 동일 패턴
- **PvP 전투력 계산에 스킬 보유효과가 여전히 미포함**(장비는 051에서 이미 반영됨) — 스킬 50종을 SQL로 포팅하려면 계산식 전체를 옮겨야 해 범위가 큼. 스킬 비중이 커지면 반영 검토 필요([`pvp.md`](./pvp.md))
- **월드보스도 동일한 "클라이언트 계산 + 서버 상한 클램프" 패턴**을 씀 — 전투는 클라이언트에서 진행되고, 종료 후 보고된 데미지를 `전투력×60`(최소 2000) 상한으로 클램프해 공유 체력에 반영(`report_world_boss_damage`). 완벽한 서버권위는 아니지만 전투력 기준 상한이 있어 터무니없는 값 보고는 막힘([`world-boss.md`](./world-boss.md))
