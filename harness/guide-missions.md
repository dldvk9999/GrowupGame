# 가이드 미션

관련 파일: `missions.js`, `MissionFloatingButton.jsx`, migration 020/021/022/029

## UI

- 화면 우하단(모바일은 하단 폭 전체)에 **항상 떠있는 플로팅 버튼**. "미션 #N", 아이콘, 라벨, 진행도(`N/M`)와 진행바 표시
- **완료되면 초록 테두리 + 펄스 애니메이션**으로 하이라이트, 클릭하면 `claim_mission_reward` 호출 → 골드 지급 + 다음 미션 자동 전환

## 일반 반복 미션 (4종 순환)

`mission_number % 4`로 순환:

| 순서 | 미션 키 | 조건 | 보상 |
|---|---|---|---|
| 1 | `kill_monsters` | 몬스터 10마리 처치 | 💰800 |
| 2 | `spend_gold` | 골드 **N** 사용(043부터 뽑기레벨에 따라 동적, 아래 참고) | 💰target의 1% |
| 3 | `login_minutes` | **60초** 접속 유지 | 💰600 |
| 4 | `use_skills` | 스킬 15회 사용 | 💰700 |

### `spend_gold` 미션 목표치 동적 조정 (migration 043, 045에서 상한 갱신)

원래 10,000골드 고정이었는데, 뽑기를 많이 한 유저에겐 100연차 한 번이면 끝날 정도로 쉬워지는 문제가 있었음.

`claim_mission_reward`가 다음 `spend_gold` 미션을 배정할 때, **스킬 뽑기레벨**(`total_skill_draws`)과 **4개 장비 슬롯 뽑기레벨 평균**을 합쳐 "종합 뽑기레벨"(1~**50**, 045에서 20→50 확장)을 구하고, 이 레벨에 따라 목표치를 **10만 골드 단위**로 계단식 상향함:

| 종합 뽑기레벨 | 목표 골드 |
|---|---|
| 1~8 | 100,000 |
| 9~20 | 200,000 |
| 21~30 | 300,000 |
| 31~43 | 400,000 |
| 44~50 (최대) | 500,000 |

보상도 목표치의 1%로 스케일링(50만 목표면 5000골드). 신규 유저는 최소 10만부터, 뽑기레벨 최대(50)면 50만까지.

(`login_minutes`의 target은 DB상 "분" 단위 1이지만 표시 라벨만 `t*60`으로 초 환산해 "60초 접속 유지하기"로 보여줌)

## 우선순위 온보딩 미션

`claim_mission_reward`가 다음 미션을 정할 때마다 재검사하며, 항상 일반 미션보다 먼저 끼어듦:

1. 활성 몬스터가 전직 가능 레벨(30/60/100/140/180)인데 아직 안 했으면 `job_tier1`~`job_tier5`(보상 3000~40000) 최우선 배정
2. 전직 조건이 없으면 스킬 슬롯이 새로 열렸는데 덜 채워졌는지(`equip_skill_slot`, 보상 1000) 확인
3. 둘 다 아니면 일반 미션으로

온보딩 미션들은 클라이언트가 보낸 진행도를 안 믿고, **완료 판정 시 서버가 `unlocked_job_tier`/`equipped_skills`를 직접 재조회해서 검증**함(진행도 카운터 우회 불가).

⚠️ **표시 버그(자체 발견/수정)**: `MissionFloatingButton.jsx`의 `CONDITION_MISSIONS`(진행도 카운터 대신 "조건 달성 시 자동완료" 문구를 보여줄 미션 키 목록)에 `job_tier1`~`job_tier4`는 있었는데 **`job_tier5`가 빠져있었음**. 5차 전직 온보딩 미션 유저는 실제 조건형 미션인데도 "0/1"이 멈춘 것처럼 표시되던 문제(실제론 전직만 하면 정상 완료됨, 표시만 오류). `job_tier5`를 Set에 추가해 수정.

## 진행도 갱신 방식

- `bumpMission(missionKey, amount)`가 `increment_mission_progress` RPC 호출 — 현재 활성 미션 키와 일치할 때만 반영, 1회 증가폭은 서버에서 최대 1000 캡(남용 방지)
- 호출 지점: 몬스터 처치(스테이지클리어/자동사냥/일반던전/전직던전 승리 4곳), 스킬 사용(3개 전투화면 공통), 골드 소비(뽑기 성공 시 소비액만큼)
- **여러 화면에서 진행도를 올려도 플로팅 버튼이 즉시 갱신**되도록 `missions.js`에 `toast.js`와 동일한 pub-sub(`subscribeMissionUpdate`)을 두고 `App.jsx`가 구독
- "N분 접속 유지"는 `App.jsx`의 1분 간격 타이머가 활성 미션이 `login_minutes`일 때만 증가

클라이언트가 진행도를 부풀려 보낼 수 있지만(`bumpMission('spend_gold', 1000)` 반복 등), 보상 자체가 소액(일반 미션 600~1200골드, `spend_gold`만 043부터 최대 5000골드, 온보딩 미션만 예외적으로 큼)이라 리스크 대비 실효성이 낮고, 온보딩 미션(가장 큰 보상)은 서버가 실제 게임 상태로 재검증하므로 조작 불가. 클레임 자체도 2초 쿨다운으로 막혀 있어 실질 파밍 속도가 제한됨.

## 버그 수정 이력

### 1. 온보딩 미션이 완료해도 버튼이 안 넘어가던 버그

`job_tier1`~`5`/`equip_skill_slot`(온보딩)은 `progress`가 절대 증가하지 않고 `target`이 항상 1로 고정인데, 클라이언트가 완료 판단을 `mission.progress >= mission.target`으로만 계산해 **실제로 조건을 채워도 버튼이 영영 안 바뀌던 버그**.

수정: `isMissionComplete(mission, {unlockedJobTier, equippedSkillCount, skillSlotLimit})`가 미션 종류별로 실제 게임 상태를 직접 보고 판정하도록 변경, `App.jsx`가 이 결과를 `completed` prop으로 명시적으로 내려줌.

**새로운 온보딩형(boolean) 미션을 추가할 땐 `isMissionComplete`에도 케이스를 같이 추가해야 함**, 안 그러면 같은 버그 재발.

### 2. 쿨다운 오탐 버그

`claim_mission_reward`의 쿨다운이 `mission_state.updated_at` 기준이었는데, `increment_mission_progress`도 진행도를 올릴 때마다 `updated_at`을 갱신해서 — **미션을 막 완료시킨 직후 완료 버튼을 눌러도 "너무 빠릅니다" 오탐**이 뜨던 문제.

수정(029): `mission_state`에 `assigned_at`(미션 배정 시각) 컬럼을 별도로 두고 쿨다운 체크는 이 값만 봄.

**진행도 갱신 시각과 쿨다운 판정 시각은 절대 같은 컬럼을 쓰면 안 됨** — 재발 방지 메모.

### 3. 큰 액수 골드 소비가 미션 진행도에 일부만 반영되던 버그

`increment_mission_progress`는 남용 방지로 1회 최대 1000까지만 반영되는데, 뽑기(`bumpMission('spend_gold', totalSpent)`)는 100연차처럼 한 번에 수만 골드를 소비해도 캡 없이 그대로 보내고 있었음(예: 3만 소비해도 미션엔 1000만 반영).

수정: `bumpMission`이 amount를 1000 단위로 쪼개서 여러 번 순차 호출(`MISSION_BUMP_CHUNK`). 활성 미션이 바뀌거나 target 도달 시 중간에 멈춤. **1000 넘는 값을 반영할 땐 항상 이 청크 분할을 거칠 것** — RPC를 직접 호출하면 같은 문제 재발.

## 보안 패치

⚠️ 처음엔 진행도 채우기+클레임 사이에 시간제한이 없어서, devtools로 `bumpMission`→`claimMissionReward`를 반복 호출하면 실제 플레이 없이 골드를 받아갈 수 있었음. `claim_mission_reward`에 **"미션 배정 시각으로부터 최소 20초"** 게이트 추가(idle 보상의 2.5초 제한과 동일 설계, 022, 029에서 기준 컬럼을 `assigned_at`으로 수정). 진행도 채우기 자체는 여전히 빠르게 가능하지만 클레임이 20초에 1번으로 막혀 파밍 속도가 제한됨.

⚠️ **081에서 20초 → 2초로 단축**(사용자 피드백) — "미션 완료 후 바로 다음으로 넘어가고 싶은데 20초 대기가 답답하다"는 요청. 매크로 방지 목적은 2초로도 어느 정도 유지되고, 진행도를 실제로 채워야 완료 판정이 나는 구조(서버가 재검증)라 클레임 간격을 줄여도 파밍 속도는 크게 안 늘어남.

## 모바일 레이아웃 겹침 주의

플로팅 버튼이 `position:fixed`라서 콘텐츠 하단(특히 전투화면 스킬버튼 줄)을 가릴 수 있음 — `app-main`의 하단 padding을 데스크톱 110px/모바일 150px로 넉넉하게 잡아서 방지(`index.css`). 하단에 새 UI 추가 시도 이 여백을 고려할 것([`ui-and-ux.md`](./ui-and-ux.md)).
