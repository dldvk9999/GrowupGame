# 캐릭터/육성/전직

관련 파일: `growth.js`, `speciesData.js`, `speciesDbIds.js`, `jobAdvancement.js`, `monsters.js`

## 스타터/진화

- 스타터 3종: `fire_1`(이모탄) / `water_1`(아쿠파피) / `grass_1`(새프링)
- 각 속성은 3단계 진화(1→2→3차): 예) 이모탄→이모드릴→이모라돈(Lv.15, Lv.30에서 자동 진화)
- **진화**와 **전직**은 별개 시스템: 진화=외형/도감상 종족 변경, 전직=같은 종족 내 강함 단계
- 9종(3속성×3진화) 전부 벡터 SVG 일러스트 완성(`assets/sprites/FireStage1~3.jsx` 등)

### 능력치 상세 카드 (`MyPage.jsx`)

마이페이지에 "기본(레벨·전직) / 장비 보너스 / 스킬 보유효과 / 최종" 4행 표로 ATK/DEF/HP 구성을 투명하게 보여줌. 전투 화면들에 내려주던 `equipmentOnlyBonus`(용의 버프 배율 미적용 순수 장비 보너스)와 `skillPossessionAtk`를 그대로 재사용. 용의 버프가 켜져있으면 표 아래에 "전투 시 20배 적용"만 별도 안내하고, 표 자체는 배율 적용 전 값으로 유지해서 "순수 강함"과 "전투 중 실제 강함"을 구분함.

### 몬스터 애칭 (migration 058)

`owned_monsters.nickname`은 001부터 있었지만 어디서도 쓰인 적 없는 죽은 컬럼이었음(재검토 중 발견). 마이페이지 "대표 몬스터" 행 아래에 애칭 편집 UI를 추가해 연결함.

- `set_monster_nickname(p_nickname)` — 1~12자(한글/영문/숫자), `null`이면 애칭 해제
- `hydrateMonster()`가 `nickname || speciesName`으로 표시 이름 결정 — 애칭을 지으면 전투 로그, 헤더 등 `player.name`을 쓰는 모든 곳에 자동 반영(`speciesName`은 별도 필드로 항상 보존)
- 계정 닉네임(`profiles.nickname`, 평생 1회만 수정)과는 완전히 별개 — 애칭은 유일성 제약도 없고 자유롭게 여러 번 변경 가능
- **랜덤 이름 추천**(신규, `lib/nameSuggestion.js`) — 애칭 입력창 옆 "🎲 추천" 버튼으로 속성(불/물/풀)에 맞는 이름 후보를 무작위 채움. 순수 클라이언트 정적 목록(속성별 8개), 서버 호출 없음

### 인기 속성 통계 (migration 068, 신규 콘텐츠)

도감 하단에 "🌍 전체 유저 계약 속성 비율" 막대그래프. `owned_monsters`가 "본인만 조회" RLS라 전체 집계는 `fetch_element_popularity()`(security definer)가 개인정보 없이 속성별 인원수/비율(%)만 반환. 활성 몬스터가 없는 유저는 `is_active = true` 필터로 자동 제외.

### 몬스터 도감 (`MonsterDex.jsx`)

마이페이지 상단의 3×3(속성×진화단계) 그리드. 서버에 "발견 기록"을 저장하지 않고 **매번 클라이언트에서 활성 몬스터의 현재 속성/단계를 기준으로 계산**함 — 내 속성과 같고 도달한 단계 이하만 스프라이트가 보이고 나머지는 실루엣(`?`) 잠김. 스타터 1종만 계약해 진화시키는 구조라 다른 두 속성은 항상 미발견 상태가 정상(포획 기능 생기기 전까지). 발견된 칸에는 `speciesData.js`의 **레벨1 기준 기본 스탯**도 표시해 다음 진화가 얼마나 강한지 가늠할 수 있게 함(실제 성장치 아님을 안내문에 명시). 별도 보상 없는 순수 수집 열람 화면.

## 레벨업/스탯 성장

- **레벨업 경험치**: `expToNextLevel(level) = round(20 * level^1.5)`
- **스탯 성장**: `base스탯 × (1 + (level-1)*0.12) × 전직배율`
- `speciesKeyToDbId`/`dbIdToSpeciesKey`(`speciesDbIds.js`)로 클라이언트 키(`fire_1` 등)와 DB `species_id`(정수)를 매핑

## 전직 시스템 (1~10차, 127에서 6~10차 확장)

- Lv.30/60/100/140/180/**240/300/360/420/480**(6~10차, 사용자 요청)에서 조건 충족 시 "✨ 전직 가능!" 배너가 뜨지만(`hasPendingJobAdvancement`), **레벨업만으로 자동 전직되지 않음** — 전직 던전([`stages-and-dungeons.md`](./stages-and-dungeons.md))을 클리어해야 실제 적용
- `owned_monsters.unlocked_job_tier`(0~10, 127에서 체크제약 확장)가 "실제 적용된" 전직 단계의 단일 진실 공급원
- `jobAdvancement.js`가 두 개념을 분리 관리: `getEligibleTierNumber(element, level)`(레벨 기준 "조건 충족", 알림용), `getAppliedTier(element, unlockedJobTier)`(실제 적용, 스탯/스킬/외형 반영) — 둘 다 `JOB_TIERS[element]` 배열 길이에만 의존하는 범용 로직이라 6~10차 추가 시 이 함수들 자체는 **손댈 필요가 없었음**(배열만 늘리면 자동으로 동작)
- 전직할 때마다 **전용 스킬 1개씩 추가 습득**(누적, 기본 뽑기 스킬과 별개) — `getJobSkillTier`의 정규식을 `job1~5`에서 `job1~10`까지 인식하도록 확장
- 전직 성공 시 **외형도 전용 그래픽으로 변경**(아래 "외형 변경" 참고)
- **최대 레벨 500** 신설(`growth.js MAX_LEVEL`, 10차 전직 요구 레벨 480보다 여유 있게 설정) — 만렙 도달 후엔 경험치가 무의미하게 계속 쌓이지 않도록 0으로 고정

### 외형 변경

- `getDisplaySpriteKey(speciesId, element, unlockedJobTier)`가 `unlockedJobTier > 0`이면 `${element}_job${tier}` 키를 반환해 전직 전용 그림 표시
- `JobTierSprite.jsx`(공용 렌더러)가 오라 크기/날개/왕관을 단계별로 화려하게 렌더링 — tier가 오를수록 오라가 커지고(tier≥2 날개, tier≥3 왕관+할로, **tier≥6 추가 백색 오라 링** 신설), 새로운 도형 없이 같은 컴포넌트가 tier 값만 받아 확장됨
- `assets/sprites/index.jsx`에 3속성×10단계 = **30개** 스프라이트 키 전부 등록 — ⚠️ 이 레지스트리는 패턴 매칭이 아니라 **키를 하나하나 직접 등록하는 정적 객체**라, 6~10차 데이터만 추가하고 여기 등록을 빠뜨렸다면 캐릭터가 빈 placeholder로 보이는 조용한 버그가 났을 것(작업 중 자체 발견/반영)

## 6~10차 전직의 추가 게이팅 (신규, 사용자 요청)

"단순히 레벨만 조건으로 있는 게 아니라" 갈수록 여러 시스템에 걸친 조건이 필요하도록 설계:

| 단계 | 추가 조건 |
|---|---|
| 6차 | 무한의 탑 20층 이상 |
| 7차 | 무한의 탑 40층 이상 + 가이드미션 15개 이상 진행 |
| 8차 | 무한의 탑 60층 이상 + 가이드미션 25개 이상 + 업적 "차원의 정복자"(`stage_clear_1000`) 보유 |
| 9차 | 무한의 탑 80층 이상 + 가이드미션 35개 이상 + 업적 "종말의 위용"(`power_1m`) 보유 |
| 10차 | 무한의 탑 100층 이상 + 가이드미션 50개 이상 + 업적 "정점의 지배자"(`level_180`) 보유 |

- **서버(`start_job_dungeon`, 127)가 최종 검증** — `tower_progress.highest_floor`, `mission_state.mission_number`, `achievement_claims`를 직접 조회해서 확인. 클라이언트(`JobDungeonPanel`)는 탑층수/미션진행은 미리 체크해서 버튼을 비활성화하지만, **업적 보유 여부는 클라이언트에서 미리 확인하지 않음**(서버 에러 메시지로만 안내) — 굳이 클라이언트에 업적 목록을 따로 긁어오는 배선을 추가하지 않고, 이미 확립된 "서버가 최종 진실"이라는 원칙에 기댐
- 판정식이 클라이언트(`JOB_DUNGEON_EXTRA_REQ`, 안내용)와 서버(`start_job_dungeon`, 실제 검증) 양쪽에 각각 있어 **하나를 바꾸면 반드시 둘 다 같이 바꿔야 함**(100/121의 "행운/요일 던전" 판정식 이중관리와 동일한 주의사항)

## 전직 던전과의 관계

- 전직 던전 클리어(`claim_job_dungeon`)가 `unlocked_job_tier`를 실제로 올리는 유일한 경로 — 이 함수 자체는 `unlocked_job_tier = v_session.tier`로 범용 대입이라 6~10차 확장 시 **손댈 필요 없었음**(체크 제약만 풀어주면 자동으로 동작)
- 던전 난이도/보스 스탯/입장 규칙은 [`stages-and-dungeons.md`](./stages-and-dungeons.md)의 "전직 던전" 참고

### 캐릭터 카드 요약 (신규 콘텐츠, `MyPage.jsx`)

마이페이지 상단에 닉네임+칭호, 대표 몬스터, 4개 핵심 지표(PvP 티어/무한의 탑 최고층/업적 개수/클리어 스테이지)를 보여주는 요약 카드. 흩어진 정보를 한 곳에 모아 진행 상황을 바로 파악 가능하게 함.

- 새 서버 호출은 업적 개수(`fetchClaimedAchievements`, 기존 함수 재사용) 하나뿐, 나머지는 이미 로드된 데이터
- 2열 그리드 레이아웃이라 넘침 위험이 낮음, 극단값(다이아몬드+9999층+29/29+1000/1000)으로 실측 검증 완료

### 성장 곡선 미리보기 (신규 콘텐츠, `MyPage.jsx`)

능력치 상세 카드 아래에 "현재 전직 단계 기준 레벨만 올랐을 때"의 예상 스탯(현재/+10/+30/+50레벨) 표. `growth.js`의 `scaleStats(species, level, unlockedJobTier)`를 그대로 재사용(이미 실전 검증된 공식) — 순수 클라이언트 계산, 서버 호출 없음.

- 장비/스킬 보너스는 의도적으로 미포함(순수 종족 성장곡선용)임을 안내문에 명시
- "현재 전직 단계 기준"이라는 전제를 명확히 표기해 미래 진화/전직에 대한 오해 방지
- 큰 숫자(백만 단위) 조합 모바일 넘침 없음 재검증
- **전투력 컬럼 추가**(사용자 경험 개선) — 각 레벨 시점의 예상 전투력(`calculateCombatPower`)도 함께 표시해 성장폭을 한눈에 비교 가능. 5개 컬럼 극단값 조합 모바일 넘침 없음 재검증 완료

### 다음 목표 위젯 (신규 콘텐츠, `lib/nextGoals.js`)

캐릭터 카드 하단에 "🎯 다음 목표" 칩 리스트 — PvP 다음 티어까지 남은 승수, 무한의 탑 다음 10층 마일스톤, 다음 출석 마일스톤(7/30/100/200), 코스튬 컬렉션 완성까지 남은 종수(20종 기준), 전투력 마일스톤(10K/100K/1M, 083 연동), 던전 깊이 마일스톤(100/300/500, 088 연동), 스킬 컬렉션(50종, 092 연동), 최고 강화수치(+1000, 091 연동) — 총 8개 목표를 순수 클라이언트 계산으로 통합(서버 호출 없음, 이미 로드된 데이터만 조합). `pvpTier.js`의 `getWinsToNextTier` 등 기존 헬퍼 재사용. 전투력은 `MyPage.jsx`의 스탯 상세 카드가 계산해두는 최종 atk/def/hp를 `calculateCombatPower`에 넣어 산출(서버 `fetch_my_combat_power()`와 정확히 같지 않을 수 있으나 순수 참고용 위젯이라 문제없음). 신규 유저가 아무 목표도 안 채웠을 때(8개 칩 동시표시) 최악의 경우까지 모바일 실측 검증 완료(넘침 없음).

### 연속 접속 스트릭 표시 (신규 콘텐츠, `lib/loginStreak.js`)

캐릭터 카드에 "🔥N일 연속 접속" 배지(2일 이상일 때만). **순수 클라이언트 동기부여용**이고 보상 계산엔 미사용 — `localStorage`로 "이 기기에서 어제도 접속했는지"만 비교해 이어가거나 리셋. 서버 검증 없는 값이라 기기 변경/데이터 삭제 시 리셋되지만, 보상과 무관한 재미 요소라 문제없음.

### 다음 레벨까지 예상 소요시간 (신규 콘텐츠, `lib/idleTimeEstimate.js`)

성장 곡선 카드에 "⏱️ 자동사냥만으로 다음 레벨까지 약 N분/시간" 추가 — 자동사냥 경험치 공식(`stages.js`의 `getIdleMonster`와 동일, 0.19 배율 반영)과 처치 텀(1.5초)을 재사용해 순수 클라이언트로 추정. 챕터는 클리어한 스테이지 수(`clearedCount / 10`)로 근사. "스테이지 도전/던전 등을 병행하면 더 빨라진다"는 문구로 "이게 최소 하한선"임을 안내. 레벨5~220 전 구간 2분~10분 수준의 합리적 범위임을 시뮬레이션 확인, 모바일 줄바꿈도 검증 완료.
