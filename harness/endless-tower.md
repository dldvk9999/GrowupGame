# 무한의 탑 (migration 071, 신규 던전 콘텐츠)

관련 파일: `lib/tower.js`, `DungeonSelect.jsx`(`TowerPanel`), `DungeonBattle.jsx`(재사용), migration 071

## 개요

기존 10층 고정 던전(경험치/골드)과 별개로, **상한 없이 계속 올라가는 도전 모드**. 한 층씩 순서대로 도전하고, 이기면 최고기록이 갱신됨. 다른 유저와 "최고 도달 층수"로 경쟁하는 랭킹도 포함.

- 던전 탭에 "🗼 무한의 탑" 신설(기존 4탭: 경험치/골드/전직던전/월드보스 + 신규 1탭)
- 하루 3회 입장 제한(기존 던전과 동일 패턴)
- 매번 "현재 최고기록 + 1"층부터 도전
- 클리어하면 그 층수가 최고기록보다 높을 때만 기록 갱신, 골드는 매번 지급(재도전이라도)

## 서버 설계 — 기존 던전 시스템과 완전히 독립

기존 `dungeon_sessions`/`dungeon_progress`/`dungeon_attempts` 테이블은 `dungeon_type in ('exp','gold')` CHECK 제약이 걸려있어서, 여기에 새 타입을 끼워 넣으려면 기존 제약을 변경해야 해서 리스크가 있었음. 대신 **완전히 독립된 새 테이블 3개**로 구현해서 기존 던전 시스템에 전혀 영향 없음:

- `tower_progress(user_id pk, highest_floor)` — 최고 기록
- `tower_attempts(user_id, attempt_date, count)` — 일일 입장 제한(기존 던전과 동일 패턴)
- `tower_sessions(id, user_id, floor, claimed, created_at)` — 도전권, `floor`는 상한 없이 계속 증가(`check (floor >= 1)`만 있음)

### 전투 판정 방식 — 기존 던전과 동일한 신뢰 모델

`claim_tower_floor`는 "세션이 유효하고 아직 클레임 안 됐으면 무조건 성공으로 신뢰"하는 구조임. **이건 새로 만든 취약점이 아니라, 이 프로젝트에 이미 확립된 기존 던전/전직던전과 동일한 패턴**(`todo.md`에 "던전/전직던전 실제로 이겼는지 미검증"으로 이미 기록된 한계)을 그대로 따른 것. 매크로 스팸을 억제하기 위해 세션 생성 후 최소 2초 경과를 요구함(기존 던전과 동일).

`enter_tower`가 "현재 최고기록+1" 층수로 세션을 발급하는 방식도 기존 `use_dungeon_attempt`(현재 클리어한 스테이지+1로 세션 발급)와 동일한 구조라, 이론적으로는 정교하게 타이밍을 맞춘 동시 요청으로 같은 층수의 세션을 여러 개 만들 수 있는 특성까지 그대로 물려받음(하루 3회 제한이 있어서 피해 규모는 제한적). 이것도 기존 시스템에 이미 있던 특성을 그대로 재사용한 것이라 새로 만든 위험은 아님 — diff 대신 010의 `use_dungeon_attempt` 로직과 나란히 비교해서 동일 패턴임을 확인함.

### 골드 보상 공식

`calc_tower_gold(floor)`가 `220 + floor^1.8 * 200`(HP 공식)을 기반으로 계산하되, **`add_gold`의 100만 상한을 넘지 않도록 `least(1000000, ...)`로 클램프**함. 시뮬레이션 검증 결과 1000층(현실적으로 도달 불가능한 극단치)에서도 정확히 100만에서 멈춤, 실제 도달 가능한 범위(수십~백 층대)에서는 자연스럽게 증가함(10층=14,123 / 30층=100,528 / 50층=251,760).

### 몬스터 난이도

클라이언트 `getTowerFloorMonster(floor)`가 서버 `calc_tower_gold`와 **동일한 HP 공식**(`220 + floor^1.8 * 200`)을 그대로 재사용해서, "몬스터가 얼마나 강한지"와 "골드를 얼마나 받는지"가 항상 일관되게 맞물리도록 함. ATK/DEF는 기존 10층 던전(`1.6/1.5/1.4` 지수)보다 더 가파른 지수(`1.8/1.7/1.6`)를 써서 "무한히 어려워지는" 긴장감을 줌. 1층(HP420)은 게임 초반 유저도 충분히 도전 가능한 수준으로 설계됨.

## 전투 화면 재사용

새 전투 컴포넌트를 따로 안 만들고 기존 `DungeonBattle.jsx`를 그대로 재사용함(`dungeonEnemy.dungeonType`이 `'exp'`/`'gold'`가 아니면 "🗼 무한의 탑"으로 표시하도록 상단 배지 텍스트만 살짝 확장). 파티클 이펙트, 스킬 사용, HP바 등 기존 전투 시스템을 그대로 활용해서 게임 프레임과 일관된 느낌을 유지함.

## 랭킹

`fetch_tower_leaderboard()`/`fetch_my_tower_rank()` — 기존 `fetch_leaderboard`/`fetch_achievement_leaderboard`와 동일한 패턴(security definer로 전체 집계, 20위 밖이면 내 순위 별도 표시). `highest_floor > 0`인 유저만 랭킹에 노출(아직 한 번도 클리어 못한 유저는 자동 제외).

## 배포 전 검증

- 자체 스캐너(ambiguous column, DROP 누락)로 5개 신규 함수 전부 통과
- 골드 공식 극단치(1000층) 시뮬레이션으로 100만 상한 클램프 확인
- Playwright로 9999층/긴 닉네임+칭호 조합 모바일 렌더링 시 가로 넘침 없음 확인

## 업데이트: 입장 제한 제거 + 보상 상향 (migration 072, 사용자 피드백)

출시 직후 사용자 피드백으로 두 가지 조정:

1. **하루 3회 입장 제한 완전 제거** — `enter_tower`를 재정의해서 `tower_attempts` 카운터 체크/증가 로직을 통째로 제거함. `tower_attempts` 테이블 자체는 남겨뒀지만(향후 재도입 가능성 대비) 더 이상 아무 함수도 참조하지 않음. 반환 컬럼 구성(`session_id, floor, remaining_attempts`)은 그대로 유지해서 DROP FUNCTION 없이 배포(반환값 중 `remaining_attempts`는 이제 의미 없는 큰 상수(999999)를 반환 — 클라이언트도 이 필드를 더 이상 안 씀). diff로 "최고층+1 계산, 세션 발급" 등 나머지 로직은 그대로 보존됨을 확인.
2. **클리어 보상 45% 상향** — `calc_tower_gold`의 배율을 `1.1` → `1.6`으로. 100만 상한은 그대로 유지, 시뮬레이션으로 극단치(1000층)에서도 상한을 넘지 않음을 재확인(10층: 14,123→20,543 / 30층: 100,528→146,223).

클라이언트(`TowerPanel`)도 "하루 3번까지" 안내 문구와 카운트다운을 제거하고 "몇 번이고 재도전할 수 있어요"로 변경, 입장 버튼의 소진 비활성화 로직도 제거함.
