# 무한의 탑 (migration 071, 신규 던전 콘텐츠)

관련 파일: `lib/tower.js`, `DungeonSelect.jsx`(`TowerPanel`), `DungeonBattle.jsx`(재사용), migration 071

## 개요

기존 10층 고정 던전(경험치/골드)과 별개로, **상한 없이 계속 올라가는 도전 모드**. 한 층씩 순서대로 도전하고 이기면 최고기록 갱신. 다른 유저와 "최고 도달 층수"로 경쟁하는 랭킹 포함.

- 던전 탭에 "🗼 무한의 탑" 신설(기존 4탭 + 신규 1탭)
- 하루 3회 입장 제한(기존 던전과 동일 패턴, **072에서 제거됨** — 아래 참고)
- 매번 "현재 최고기록 + 1"층부터 도전
- 클리어하면 기록보다 높을 때만 갱신, 골드는 매번 지급(재도전이라도)

## 서버 설계 — 기존 던전 시스템과 완전히 독립

기존 `dungeon_sessions`/`dungeon_progress`/`dungeon_attempts`는 `dungeon_type in ('exp','gold')` CHECK가 있어서 새 타입을 끼워 넣으려면 기존 제약 변경이 필요해 리스크가 있었음. 대신 **완전히 독립된 새 테이블 3개**로 구현해서 기존 던전에 전혀 영향 없음:

- `tower_progress(user_id pk, highest_floor)` — 최고 기록
- `tower_attempts(user_id, attempt_date, count)` — 일일 입장 제한(기존 던전과 동일 패턴)
- `tower_sessions(id, user_id, floor, claimed, created_at)` — 도전권, `floor`는 `check (floor >= 1)`만 있고 상한 없음

### 전투 판정 방식 — 기존 던전과 동일한 신뢰 모델

`claim_tower_floor`는 "세션이 유효하고 아직 미클레임이면 무조건 성공"으로 신뢰함. **새로 만든 취약점이 아니라 기존 던전/전직던전과 동일한 패턴**(`todo.md`에 이미 기록된 한계). 매크로 스팸 억제를 위해 세션 생성 후 최소 2초 경과를 요구(기존 던전과 동일).

`enter_tower`가 "현재 최고기록+1" 층수로 세션을 발급하는 방식도 기존 `use_dungeon_attempt`와 동일 구조라, 정교한 동시 요청으로 같은 층수 세션을 여러 개 만들 수 있는 특성까지 물려받음(하루 3회 제한이 있어 피해는 제한적). 010의 `use_dungeon_attempt`와 나란히 비교해 동일 패턴임을 확인함.

### 골드 보상 공식

`calc_tower_gold(floor)`가 `220 + floor^1.8 * 200`(HP 공식) 기반이되, **`least(1000000, ...)`로 100만 상한 클램프**. 1000층(도달 불가능한 극단치)에서도 정확히 100만에서 멈춤을 확인, 실제 도달 범위(수십~백 층대)에서는 자연스럽게 증가(10층=14,123 / 30층=100,528 / 50층=251,760).

### 몬스터 난이도

클라이언트 `getTowerFloorMonster(floor)`가 서버 `calc_tower_gold`와 **동일한 HP 공식**을 재사용해서, "몬스터 강도"와 "골드량"이 항상 일관되게 맞물림. ATK/DEF는 기존 10층 던전(`1.6/1.5/1.4` 지수)보다 가파른 지수(`1.8/1.7/1.6`)로 "무한히 어려워지는" 긴장감을 줌. 1층(HP420)은 초반 유저도 도전 가능한 수준.

### 다음 상대 미리보기 (신규 콘텐츠)

"👁️ 다음 상대 미리보기"로 다음 층 몬스터 HP/ATK/DEF를 미리 보여줌 — 무제한 입장(072)이라 부담없이 도전 가능하지만, 전투력과 비교해 승산을 가늠하도록 돕는 목적. `getTowerFloorMonster(floor)`를 그대로 재사용(전투 화면과 100% 동일 값). 천만 단위 극단값에서도 모바일 넘침 없음 확인.

### 업적 마일스톤 진행률 안내 (신규 콘텐츠)

최고 기록 카드에 "🏅 다음 업적까지 N층 남음"(10/30/100층 업적, 074/097 연동). 이미 있는 `highestFloor`만 계산, 새 서버 호출 없음.

## 전투 화면 재사용

새 전투 컴포넌트 없이 기존 `DungeonBattle.jsx`를 재사용(`dungeonEnemy.dungeonType`이 `exp`/`gold`가 아니면 "🗼 무한의 탑" 배지 텍스트만 확장). 파티클/스킬/HP바 등 기존 전투 시스템을 그대로 활용해 일관된 느낌 유지.

## 랭킹

`fetch_tower_leaderboard()`/`fetch_my_tower_rank()` — 기존 랭킹 함수들과 동일 패턴(security definer 전체 집계, 20위 밖이면 별도 표시). `highest_floor > 0`인 유저만 노출.

## 배포 전 검증

- 자체 스캐너(ambiguous column, DROP 누락)로 5개 신규 함수 전부 통과
- 골드 공식 극단치(1000층) 시뮬레이션으로 100만 상한 확인
- Playwright로 9999층/긴 닉네임+칭호 조합 모바일 가로 넘침 없음 확인

## 업데이트: 입장 제한 제거 + 보상 상향 (migration 072, 사용자 피드백)

1. **하루 3회 입장 제한 완전 제거** — `enter_tower` 재정의로 `tower_attempts` 카운터 로직 통째 제거. 테이블은 남겨뒀지만(재도입 대비) 참조 안 함. 반환 컬럼 구성(`session_id, floor, remaining_attempts`) 그대로라 DROP 없이 배포(`remaining_attempts`는 이제 무의미한 999999 상수). diff로 "최고층+1 계산, 세션 발급" 등 나머지 로직 보존 확인
2. **클리어 보상 45% 상향** — `calc_tower_gold` 배율 `1.1` → `1.6`. 100만 상한 그대로 유지, 극단치에서도 안 넘음을 재확인(10층: 14,123→20,543 / 30층: 100,528→146,223)

클라이언트도 "하루 3번까지" 문구/카운트다운 제거, "몇 번이고 재도전할 수 있어요"로 변경.

## ⚠️ 보안 수정: 중복 세션 파밍 방지 (migration 073)

072 직후 재검토 중 발견: 클레임 전에 `enter_tower()`를 빠르게 여러 번 호출하면 **같은 층수를 가리키는 미클레임 세션이 여러 개** 생김. `claim_tower_floor`는 신기록 여부와 무관하게 클레임마다 골드를 지급하므로, 중복 세션을 전부 클레임하면 같은 층수로 골드를 반복 수령하는 파밍이 가능했음. 하루 3회 제한이 이 피해를 암묵적으로 제한했었는데, 072로 무제한이 되며 이론상 무한 반복 가능해짐.

**수정**: `enter_tower()`를 멱등하게 재작성 — 이미 미클레임 세션이 있으면 새로 안 만들고 **기존 세션을 그대로 반환**. 유저당 최대 1개 미클레임 세션만 존재해 파밍 원천 차단. 부수적으로 "패배해서 나갔다가 재도전"도 같은 세션을 자연스럽게 이어씀. 반환 컬럼 그대로라 DROP 불필요, diff로 나머지 로직 보존 확인.

## 10층 단위 마일스톤 보너스 (migration 075, 신규 콘텐츠)

10/20/30...층을 **신기록으로 처음** 돌파할 때마다 `층수×800`(100만 상한) 골드 축하 우편 자동 발송(챕터/던전 완주 보너스와 동일 패턴). `mails.source_key`(`tower_milestone_층수`)로 중복 방지. `claim_tower_floor` 재정의 시 diff로 세션검증/2초체크/골드지급 등 핵심 로직 보존 확인, DROP 불필요. 클라이언트는 10층 단위 신기록이면 신기록 토스트 대신 "우편함에 보너스 도착" 안내로 전환.

## 관련 업적 (migration 074, 097)

"탑의 초입"(10층, 4000골드) / "구름 위 수련자"(30층, 15000골드) / "천공의 도전자"(100층, 60000골드, 097 추가) — CASE만 추가하는 재정의(반환타입 그대로, DROP 불필요). `tower_progress.highest_floor` 직접 조회로 판정. 45개 업적 키 diff 완전 일치 재검증 완료.

## ⚠️ 긴급 버그 수정: "column reference floor is ambiguous" (migration 078)

**증상**: 무한의 탑 입장 시 이 에러로 입장 자체가 안 됨.

**원인**: 073에서 추가한 중복세션 방지 로직(`select id, floor into v_existing from tower_sessions where ...`)이 별칭 없이 `floor`를 썼는데, `enter_tower()`의 `RETURNS TABLE(..., floor integer, ...)` OUT 파라미터명도 `floor`라서 모호해 배포 즉시 크래시. **harness에 이미 "재발 패턴"으로 기록된 것과 정확히 같은 버그**(046/056)인데 073을 급하게 작성하며 그 교훈을 다시 놓침.

**수정**: `tower_sessions`에 별칭(`ts`)을 붙여 `select ts.id, ts.floor into ... from tower_sessions ts where ts.user_id = ... and ts.claimed = false`로 명시적 구분(078). 반환 컬럼 그대로라 DROP 불필요.

**자동 스캐너 커버리지 개선**: 기존 스캐너는 `UPDATE ... SET col = col`만 검사했고, 이번 실제 패턴인 `SELECT col INTO ... FROM table`(별칭 없음)은 대상이 아니어서 073 배포 전에 못 잡음. `SELECT ... INTO`도 검사하도록 확장해서 073의 버그를 재현 검출 확인 — 다만 이 확장 스캐너는 **주석에 코드를 인용하면 그 텍스트를 실제 코드로 오인식**하는 한계가 있음을 078 검증 중 발견(078 자체는 이미 별칭이 붙어 안전한데, 주석의 073 인용문이 오탐 유발). **향후 마이그레이션 주석에서 예전 버그 코드를 인용할 땐 SQL 형태 그대로 쓰지 않는 게 안전함**.

**교훈**: 이미 문서화된 "재발 패턴"도 급하게 작성하면 다시 놓칠 수 있음. `RETURNS TABLE`이 있는 함수에 새 `SELECT ... INTO`를 추가할 땐 **항상 테이블 별칭을 습관적으로 붙이는 것**이 가장 확실한 예방책.
